## Brief

Build a scenario-based k6 suite against
[JSONPlaceholder](https://jsonplaceholder.typicode.com) that models two
concurrent user workloads against a public REST API, enforces per-scenario
thresholds, and runs in CI so threshold failures break the build.

JSONPlaceholder is a well-known, low-ceremony practice target: it accepts any
REST verb and returns plausible JSON without requiring auth. This makes it ideal
for focusing entirely on scenario architecture and CI integration — not on
token management or environment setup.

> **Drift notice.** JSONPlaceholder is hosted by typicode and has been reliably
> available for years. It does not persist writes (POST/PUT/DELETE return success
> responses but no data is actually stored). This is fine for load testing — you
> are measuring response time and error rate, not data integrity. If you need a
> fully offline target, substitute a local JSON Server instance:
> `npx json-server --watch db.json` where `db.json` provides your seed data.

## Suggested approach

1. Start from a clean k6 project.
2. Define two scenarios in the `options.scenarios` object. Recommended pair:
   - `read_posts` — a `constant-vus` executor (10 VUs, 45s) cycling through
     `GET /posts`, `GET /posts/:id`, and `GET /posts/:id/comments` — simulates
     sustained read traffic across a content feed.
   - `write_posts` — a `ramping-vus` executor (0→15→0 VUs over 1 min) hitting
     `POST /posts` and `PUT /posts/:id` — simulates a burst of content authoring.
3. Tag each request group with `{ tags: { scenario: __ENV.K6_SCENARIO_NAME } }`.
4. Define per-scenario thresholds:
   ```js
   'http_req_duration{scenario:read_posts}': ['p(95)<500'],
   'http_req_duration{scenario:write_posts}': ['p(99)<800'],
   'http_req_failed{scenario:read_posts}': ['rate<0.01'],
   'http_req_failed{scenario:write_posts}': ['rate<0.01'],
   ```
5. Add a `.github/workflows/k6.yml` that:
   - Runs `k6 run --summary-export=summary.json script.js`
   - Uploads `summary.json` as an artifact with `if: always()`
   - Fails the job when k6 exits non-zero
6. Verify the gate works: temporarily tighten one threshold to `p(95)<1` and confirm CI goes red. Revert and note this in your README.

## What "done" looks like

A public GitHub repo with:
- A `script.js` with two named scenarios, per-scenario tags, and per-scenario thresholds
- A `.github/workflows/k6.yml` that fails on breach and uploads `summary.json`
- A short README paragraph explaining the read vs write threshold difference — why different percentiles and budgets for each scenario
- A green CI badge in the README

## Why this matters

Real APIs are rarely hammered by one traffic shape at a time. A content platform
sees reads and writes concurrently; your suite needs to reflect that. Modelling
both in a single run — with separate thresholds — is how you detect that the
write path is 60% slower than the read path before your traffic spike does.
