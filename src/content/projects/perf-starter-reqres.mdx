## Brief

Install k6 from scratch and run your first real load test against
[ReqRes](https://reqres.in) — a hosted API simulator that returns realistic
JSON responses for user management operations. You'll measure how a
read-heavy versus a write-heavy endpoint behaves under the same load profile.

The lesson teaches you how k6 works. This project makes you use it on a real
HTTP target and compare the latency characteristics of a GET versus a POST.

> **Drift notice.** ReqRes is a public hosted service. Response times may vary
> depending on server load at the time you run your test. If the service is
> unavailable or significantly slower than the thresholds suggested below, note
> the observed behaviour in your README and adjust your threshold values with a
> written justification. ReqRes does not have an official local Docker image;
> if you prefer a fully offline target, substitute QuickPizza
> (`docker run -p 3333:3333 grafana/quickpizza-local`) and update the endpoints
> accordingly.

## Suggested approach

1. Install k6: `brew install k6` (macOS), `choco install k6` (Windows), or via the [official install docs](https://k6.io/docs/get-started/installation/).
2. Create `script.js` with a `baseURL` of `https://reqres.in`.
3. Write a `default` function that calls at least two endpoints. Suggested pair:
   - `GET /api/users?page=1` — retrieves a paginated list of users
   - `POST /api/users` with a JSON body (name, job) — creates a new user record
4. Add a `sleep(1)` between iterations.
5. Define an `options` block with:
   - `vus: 5` and `duration: '30s'`
   - A threshold: `http_req_failed: ['rate<0.01']`
   - A threshold: `http_req_duration: ['p(95)<800']` (ReqRes is fast but shared — 800 ms gives you headroom)
6. Run `k6 run script.js` and paste the summary table into your README.

## What "done" looks like

A folder (no public repo required at this tier) that:
- contains a `README.md` listing the k6 version used
- has a script that passes its thresholds consistently
- has a README with a one-command run instruction, the threshold definitions, and a note on whether the GET or POST endpoint was slower and what you think caused the difference

## Why this matters

Comparing GET versus POST latency under the same VU load is the first time you
will see that different verbs hit different code paths — and that your threshold
needs to account for that. Setting `p(95)<800` is a starting point; explaining
in your README whether the GET or POST drove that number is the reasoning skill
that underpins every SLO conversation you'll have in production.
