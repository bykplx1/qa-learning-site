## Core Idea

Quality Assurance is not a phase at the end of development — it is a discipline of asking "what could go wrong?" at every step. A QA mindset means treating software as a set of assumptions that need evidence, not a set of features that need checking.

> The QA mindset is evidence-based skepticism applied continuously: every claim about behaviour is a hypothesis until a test confirms or refutes it.

## Worked Example

A team ships a login feature. A developer writes a passing unit test:

```ts
expect(login('alice', 'correct-password')).toBe(true);
```

A QA engineer with the right mindset asks: what happens with an empty password? SQL injection in the username? 429 after 100 rapid attempts? The unit test confirms one happy path; the QA mindset maps the boundary conditions and threat surface the developer did not think to test.

The deliberate mistake: testing only the happy path and calling the feature "done."

## Common Pitfalls

- **Testing only happy paths.** Most bugs live at boundaries and error paths. Fix: write the unhappy path test first.
- **Late-stage quality gates.** Integrating QA only at the end of a sprint means defects compound. Fix: embed testing in Definition of Done at the story level.
- **Confusing test coverage with quality.** 90% line coverage can coexist with critical logic gaps. Fix: measure defect escape rate alongside coverage.
- **Treating QA as a separate team responsibility.** When only dedicated QA engineers think about quality, developers ship assumptions. Fix: make testability a shared engineering standard.
- **Skipping risk analysis.** Not every bug matters equally. Fix: triage by impact × probability before deciding what to automate.
