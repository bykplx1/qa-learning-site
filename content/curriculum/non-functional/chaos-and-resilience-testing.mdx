## Core Idea

Chaos engineering is **the practice of deliberately injecting failure into a running system to learn how it behaves under stress — before production answers that question unprompted.** It is not "randomly break things"; it is hypothesis-driven experimentation with a controlled blast radius, following the same scientific method applied to operational failure. Toxiproxy is the tool that makes this practice concrete at development and CI scale: it sits as a transparent TCP proxy between your application and any dependency, and lets you inject latency, timeouts, packet loss, or connection resets with a single CLI command or API call. The load-bearing claim: **a system's resilience is whatever you have evidence for, no more.** "We have circuit-breaker protection" is a claim; a Toxiproxy experiment is the test that converts the claim into evidence — or into a bug report.

> The resilience you have claimed but never tested is hope. The Toxiproxy experiment converts hope into evidence — or into a backlog item.

## 1. Set up properly

Everything from an empty folder to a single green fault-injection run against a local toy target.

**Prerequisites:** Node ≥ 22.12.0, a terminal, Docker (optional but easiest for the server binary).

### Create the project

```bash
mkdir chaos-runbook && cd chaos-runbook
npm init -y
```

### Install the Node client (pinned)

```bash
npm install --save-dev toxiproxy-node-client@4.1.0
```

### Download and run the Toxiproxy server binary

Toxiproxy is a standalone Go binary. The easiest zero-dependency approach is Docker:

```bash
docker run -d \
  --name toxiproxy \
  -p 8474:8474 \
  -p 5433:5433 \
  ghcr.io/shopify/toxiproxy:2.12.0
```

Port `8474` is the Toxiproxy control API. Port `5433` is where we will listen for proxied connections (we forward it to a local target).

Alternatively, download the binary directly (Linux x86-64 shown):

```bash
curl -Lo toxiproxy-server \
  https://github.com/Shopify/toxiproxy/releases/download/v2.12.0/toxiproxy-server-linux-amd64
chmod +x toxiproxy-server
./toxiproxy-server &
```

> Pin the version. Toxiproxy's HTTP control API occasionally gains new toxic types or renames attributes between minor releases. Only bump deliberately and re-run your integration tests in the same PR.

### Set up a local toy target to inject faults against

We need a simple TCP server to act as the downstream dependency. Create `toy-server.js`:

```js
// toy-server.js — a minimal echo server simulating a database or API dependency
import net from 'node:net';

const server = net.createServer((socket) => {
  socket.on('data', (data) => {
    // simulate a tiny processing delay, then echo back
    setTimeout(() => socket.write(`OK: ${data}`), 10);
  });
});

server.listen(5555, () => console.log('toy-server listening on :5555'));
```

Run it:

```bash
node toy-server.js
```

Expected output:

```
toy-server listening on :5555
```

### Create a Toxiproxy proxy in front of the toy target

Create `setup-proxy.js`:

```js
import { Toxiproxy } from 'toxiproxy-node-client';

const toxiproxy = new Toxiproxy('http://localhost:8474');
const proxy = await toxiproxy.createProxy({
  name: 'toy',
  listen: '0.0.0.0:5433',
  upstream: 'localhost:5555',
});
console.log('proxy created:', proxy.name);
```

```bash
node setup-proxy.js
```

Expected output:

```
proxy created: toy
```

### Inject a timeout toxic and observe the system degrade

Create `inject-timeout.js`:

```js
import { Toxiproxy } from 'toxiproxy-node-client';
import net from 'node:net';

const toxiproxy = new Toxiproxy('http://localhost:8474');

function sendRequest(port, message, timeoutMs = 3000) {
  return new Promise((resolve) => {
    const socket = net.createConnection({ port }, () => socket.write(message));
    const timer = setTimeout(() => {
      socket.destroy();
      resolve({ ok: false, error: 'timeout' });
    }, timeoutMs);
    socket.on('data', (data) => {
      clearTimeout(timer);
      socket.end();
      resolve({ ok: true, response: data.toString() });
    });
    socket.on('error', (err) => {
      clearTimeout(timer);
      resolve({ ok: false, error: err.message });
    });
  });
}

// baseline: no toxic
console.log('--- baseline (no toxic) ---');
const baseline = await sendRequest(5433, 'hello');
console.log('result:', baseline);

// inject a 2 s timeout toxic
const proxies = await toxiproxy.getProxies();
const toy = proxies.toy;
await toy.addToxic({
  type: 'timeout',
  name: 'db_timeout',
  toxicity: 1.0,           // 100% of connections affected
  attributes: { timeout: 2000 },  // ms before connection is killed
});
console.log('\n--- with timeout toxic (2 s) ---');
const degraded = await sendRequest(5433, 'hello', 3000);
console.log('result:', degraded);

// remove the toxic — system recovers
await toy.removeToxic('db_timeout');
console.log('\n--- toxic removed: recovered ---');
const recovered = await sendRequest(5433, 'hello');
console.log('result:', recovered);
```

Run to green:

```bash
node inject-timeout.js
```

Expected output:

```
--- baseline (no toxic) ---
result: { ok: true, response: 'OK: hello' }

--- with timeout toxic (2 s) ---
result: { ok: false, error: 'timeout' }

--- toxic removed: recovered ---
result: { ok: true, response: 'OK: hello' }
```

When the toxic is active the caller receives a structured error (not an unhandled crash). When the toxic is removed the system recovers cleanly. That is the green run: fault injected, degradation observed, recovery confirmed.

## 2. Implement + best practice

### Toxics reference — the four you will use most

```js
// latency — adds delay + jitter (simulates slow network or overloaded dependency)
await proxy.addToxic({
  type: 'latency',
  name: 'slow_db',
  toxicity: 1.0,
  attributes: { latency: 500, jitter: 100 },  // ms
});

// bandwidth — throttles to N KB/s (simulates congested link)
await proxy.addToxic({
  type: 'bandwidth',
  name: 'throttled',
  toxicity: 1.0,
  attributes: { rate: 10 },  // KB/s
});

// timeout — kills the connection after N ms (simulates TCP hang)
await proxy.addToxic({
  type: 'timeout',
  name: 'db_timeout',
  toxicity: 1.0,
  attributes: { timeout: 2000 },
});

// reset_peer — sends TCP RST immediately (simulates hard crash or firewall drop)
await proxy.addToxic({
  type: 'reset_peer',
  name: 'hard_crash',
  toxicity: 1.0,
  attributes: { timeout: 0 },
});
```

### Chaos-in-CI: Toxiproxy integration test

The same failure class caught locally can be gated at PR time. The pattern: spin up Toxiproxy in `beforeAll`, create a proxy, inject a toxic per test, assert the application's structured error response.

```ts
// tests/integration/resilience.test.ts
import { describe, it, beforeAll, afterAll, afterEach } from 'vitest';
import { Toxiproxy } from 'toxiproxy-node-client';

const toxiproxy = new Toxiproxy('http://localhost:8474');
let proxy: Awaited<ReturnType<typeof toxiproxy.createProxy>>;

beforeAll(async () => {
  proxy = await toxiproxy.createProxy({
    name: 'db_test',
    listen: '0.0.0.0:5433',
    upstream: `localhost:${process.env.DB_PORT ?? 5555}`,
  });
});

afterAll(async () => {
  await proxy.remove();
});

afterEach(async () => {
  // clean slate between tests — remove all toxics
  const toxics = await proxy.getToxics();
  await Promise.all(toxics.map((t) => t.remove()));
});

describe('progress API resilience', () => {
  it('returns 503 JSON when DB times out', async () => {
    await proxy.addToxic({
      type: 'timeout',
      name: 'db_timeout',
      toxicity: 1.0,
      attributes: { timeout: 100 },
    });

    const res = await fetch('http://localhost:4321/api/progress');
    expect(res.status).toBe(503);
    const body = await res.json();
    expect(body).toHaveProperty('error', 'service_unavailable');
  });

  it('recovers after the toxic is removed', async () => {
    await proxy.addToxic({
      type: 'timeout',
      name: 'db_timeout',
      toxicity: 1.0,
      attributes: { timeout: 100 },
    });
    const degraded = await fetch('http://localhost:4321/api/progress');
    expect(degraded.status).toBe(503);

    // remove toxic — circuit should close on next request
    const toxics = await proxy.getToxics();
    await Promise.all(toxics.map((t) => t.remove()));

    const recovered = await fetch('http://localhost:4321/api/progress');
    expect(recovered.status).toBe(200);
  });
});
```

This is the **chaos-in-CI** pattern: Toxiproxy in an integration test suite is the shift-left form of chaos engineering (back-reference: [[shift-left-and-shift-right]]). The GameDay confirms the same class of failure at production scale — it does not replace the integration test.

### Partial toxicity — test probabilistic failures

Set `toxicity` below `1.0` to simulate intermittent failures (the hardest class to diagnose in production):

```js
// 30% of connections get a timeout — rest pass through normally
await proxy.addToxic({
  type: 'timeout',
  name: 'intermittent',
  toxicity: 0.3,
  attributes: { timeout: 5000 },
});
```

Run the test suite under partial toxicity to find code paths that assume 100% upstream reliability.

### Backup-restore drill — the most-skipped experiment

A team claims "RTO for database failure is 2 hours." They have never tested it. The drill:

```
1. Take a full backup of the staging database (record timestamp → RPO baseline)
2. Drop all tables in staging:
     DROP SCHEMA public CASCADE; CREATE SCHEMA public;
3. Start timer
4. Restore from backup (follow your documented runbook exactly)
5. Validate data integrity: row counts, spot-checks on recent records
6. Stop timer → RTO measurement
7. Record delta between backup timestamp and first restored record → RPO measurement
```

Common findings on first drill: the restore runbook references a tool version upgraded 6 months ago, the backup download takes 40 minutes (not "a few minutes"), and one table was excluded by a misconfigured filter. The Schrödinger backup is the one that hasn't been tested for restore. Run drills quarterly; the skills decay.

## 3. Common pitfalls

- **No defined steady state before the experiment.** Without a quantified, observable steady state ("auth p95 < 500 ms, error rate < 0.1%"), the question "did the experiment fail?" is unanswerable. Fix: write down the steady state — as observable metrics with thresholds, not prose — before designing the experiment. The steady state is the oracle (back-reference: [[test-oracles-and-prioritization]]).

- **Skipping blast-radius design.** A chaos experiment that takes down production for all users is worse than the failure it was testing against. Fix: start with the smallest possible scope (one service instance, one region, 1% of traffic in staging) and have an explicit kill switch rehearsed before the experiment starts: `await proxy.removeToxic('name')` restores service instantly. Expand scope only after the small version passes.

- **Running chaos without observability.** If you cannot observe the system during the experiment you cannot detect steady-state deviation, cannot diagnose causes, and cannot tell when recovery completes. Fix: build [[observability-for-testers]] first; chaos is the discipline observability unlocks, not the starting point.

- **Findings that die in Slack.** An experiment that surfaces a retry storm or a missing circuit breaker, then records the finding in a Slack thread that nobody follows up on, makes chaos theatre rather than discipline. Fix: every finding becomes a dated backlog item (code fix, runbook update, new alert, or new test). Track to completion.

- **Chaos as a one-time event.** A single GameDay is not a chaos programme. Resilience decays as infrastructure and dependencies change. Fix: schedule recurring GameDays (quarterly is a common cadence); treat the schedule as a recurring calendar commitment, not a project milestone.

- **Retries without backoff, jitter, or a budget.** Synchronised retries amplify load on a failing dependency — a 3× retry multiplier turns 1 RPS of failures into 4 RPS, potentially re-saturating a dependency that was recovering. Fix: exponential backoff, full jitter, and a retry budget on every external call. Toxiproxy's partial-toxicity mode is exactly the experiment to verify the mitigation is implemented.

- **Backup-restore never drilled.** A backup that has never been restored is a hypothesis, not a recovery mechanism. Fix: run a restore drill quarterly against a non-production environment. Time the restore; validate the data; update the runbook.

## 4. Maintain

<div role="list" aria-label="Maintenance triggers and responses">
<Maintain trigger="Toxiproxy ships a new release (e.g. v2.x → v2.y)">
  1. Read the changelog for breaking API changes (the HTTP control API occasionally gains new toxic types or renames attributes).
  2. Update the Docker image tag in your `docker run` command or CI service definition.
  3. Update `toxiproxy-node-client` if the new server version requires a newer client API.
  4. Re-run all integration tests that use Toxiproxy — confirm no attribute-name regressions.
  5. Update the `verified.versions` block and `verified.date` in frontmatter.
</Maintain>

<Maintain trigger="An integration test using Toxiproxy hangs indefinitely in CI">
  1. Check that the Toxiproxy server container started before the test runner (`depends_on` or health-check in CI service config).
  2. Verify the proxy port (`5433` in examples) is not already in use by another service — Toxiproxy fails silently on port conflict.
  3. Check `afterEach` cleanup: a toxic left over from a previous test can hold a connection open and block the next test.
  4. Add a hard timeout to the fetch/socket call in the test (`AbortSignal.timeout(3000)`) so a hung test fails fast rather than blocking the whole suite.
</Maintain>

<Maintain trigger="A chaos experiment finds a retry storm in production (circuit breaker not wired)">
  1. Add the circuit breaker in code (or enable it in your service mesh or HTTP client library).
  2. Write a Toxiproxy integration test that injects the failure and asserts the circuit-breaker fast-fails within the expected number of attempts.
  3. Add the integration test to CI so the same failure class is gated at PR time.
  4. Add a GameDay scenario for this failure class at the next quarterly drill.
  5. Update the runbook entry in your on-call documentation.
</Maintain>

<Maintain trigger="RTO/RPO drill shows recovery takes longer than the objective">
  1. Identify the bottleneck step in the restore runbook (download time, tool version mismatch, missing credentials).
  2. Fix the bottleneck: automate the step, pre-stage the backup file, or update the runbook to reflect the real tool versions.
  3. Re-run the drill immediately after the fix — do not defer to the next quarterly cycle.
  4. If the objective itself is unachievable, escalate to revise the SLO before the next incident.
  5. Record both the pre-fix and post-fix RTO measurements in the drill log.
</Maintain>

<Maintain trigger="GameDay finding is filed but never acted on (finding rots in backlog)">
  1. Assign a DRI (directly responsible individual) to every finding at the end of the GameDay debrief — not "the team."
  2. Set a due date: code fixes within one sprint; runbook updates within the same sprint; new monitoring alerts before the next GameDay.
  3. Open the finding as a real issue (not a Slack thread) with a checklist: code fix, test added, runbook updated, alert configured.
  4. Review the finding backlog at the start of the next GameDay preparation — close or re-prioritise any stale items before running new experiments.
</Maintain>
</div>

## Practice Task

<PracticeTask id="chaos-task-1" rubric="chaos-rubric-v1">
  Design and execute a chaos experiment against a local toy target using Toxiproxy. A chaos experiment is not "let's see what happens" — it is hypothesis-driven: define steady state, form a falsifiable hypothesis, inject the fault, observe, harden.

  **Part 1 — Set up Toxiproxy and a toy target**

  Follow the Set up section to get a green baseline run: toy server responding, proxy created, baseline request succeeds. Capture the terminal output showing the green baseline.

  **Part 2 — Inject a timeout toxic and observe**

  Inject a 2-second timeout toxic against the proxy. Send 5 requests through the proxy with a 3-second caller timeout. Record:
  - What fraction of requests timed out?
  - What error did the caller receive?
  - Did the caller crash, or did it return a structured error?

  The hypothesis to test: "When the upstream times out, the caller returns a structured error within the caller-timeout window and does not crash." Did the hypothesis hold?

  **Part 3 — Inject partial toxicity**

  Set `toxicity` to `0.3` (30% of connections get a timeout). Send 20 requests and record the pass/fail distribution. State what production failure class this simulates and why it is harder to debug than 100% failure.

  **Part 4 — Harden: write the integration test**

  Write a Vitest integration test (following the chaos-in-CI pattern from section 2) that:
  - Injects a timeout toxic.
  - Asserts the application returns a structured error (not an unhandled exception).
  - Removes the toxic in `afterEach` so later tests start clean.

  The test does not need to run against the real application — it can test a simple wrapper function that sends a request through the proxy and returns `{ ok: false, error: 'timeout' }` on failure.

  **Part 5 — GameDay plan (one scenario)**

  Write a single GameDay scenario for the toy target (or a system you have access to):
  - Steady state: at least 2 observable metrics with quantified thresholds.
  - Hypothesis: falsifiable prediction — not "let's see what happens."
  - Blast radius: the smallest scope that tests the hypothesis (local, staging, 1% traffic).
  - Kill switch: the exact Toxiproxy API call that restores service.
  - Abort condition: the threshold at which you would abort the experiment.

  Rubric (revealed after submission):
  - Does Part 2 record the error type — not just "it failed"? "Timeout" is not enough; name whether it was a connection error, a timeout, or an unhandled crash.
  - Does Part 3 name the production failure class (intermittent dependency failure) and explain why 30% failure is harder to reproduce than 100%?
  - Does Part 4's integration test clean up all toxics in `afterEach` — not just the one it added? Leaving toxics pollutes later tests.
  - Does the GameDay scenario have an explicit, falsifiable hypothesis — not "let's see what happens"? Unfalsifiable hypotheses fail the rubric.
  - Does the GameDay scenario name its kill switch as a specific Toxiproxy API call — not just "remove the toxic"?
</PracticeTask>
