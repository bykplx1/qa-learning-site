## Core Idea

Observability is the property of a system that lets you ask any question about its current and past behaviour **without shipping new code**. It is enabled by three primary signals — *logs, metrics, traces* — and consumed at the moments your test suite cannot help you: after deployment, under real load, with real users. For the tester, observability is the production-truth feedback loop the test suite cannot provide. A bug report that says "checkout is slow sometimes" is a wild goose chase without telemetry; paired with a trace and a Grafana panel it becomes "p99 latency spiked at 14:32, caused by a payment-service retry storm introduced in the last deploy." OpenTelemetry (OTel) is the CNCF-standard instrumentation layer that emits all three signals from your application code into any compliant backend — Jaeger, Prometheus, Grafana, Datadog, Honeycomb — without vendor lock-in.

> Observability is not a replacement for testing. It is the production oracle that proves whether the thing you tested actually holds at scale.

## 1. Set up

Everything from an empty folder to a green OTel trace landing in a local Jaeger UI.

**Prerequisites:** Node ≥ 22.12.0, Docker (for the Jaeger all-in-one collector), a terminal.

### Create the project

```bash
mkdir otel-runbook && cd otel-runbook
npm init -y
```

### Start Jaeger (local trace backend)

Jaeger's all-in-one image ships an OTLP receiver, a query API, and a UI in one container.

```bash
docker run --rm -d \
  --name jaeger \
  -p 4318:4318 \
  -p 16686:16686 \
  jaegertracing/all-in-one:1.65
```

Port 4318 receives OTLP/HTTP traces. Port 16686 serves the Jaeger UI — open it at http://localhost:16686 once the container is running.

### Install the OTel SDK (pinned)

```bash
npm install \
  @opentelemetry/sdk-node@0.57.0 \
  @opentelemetry/auto-instrumentations-node@0.56.0 \
  @opentelemetry/exporter-trace-otlp-http@0.57.0 \
  @opentelemetry/api@1.9.0
```

> Pin every OTel package to the same minor release family. The `@opentelemetry/api` package is the stable surface; `sdk-node` and the exporters must share the same semver minor to avoid peer-dep conflicts.

### Write the instrumentation initialiser

Create `src/instrumentation.js` — this file must be loaded **before any application code**:

```js
// src/instrumentation.js
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';

const sdk = new NodeSDK({
  serviceName: 'checkout-service',
  traceExporter: new OTLPTraceExporter({
    url: 'http://localhost:4318/v1/traces',
  }),
  instrumentations: [getNodeAutoInstrumentations()],
});

sdk.start();

process.on('SIGTERM', () => sdk.shutdown());
```

### Write a minimal app that emits a trace

Create `src/app.js`:

```js
// src/app.js
import { trace } from '@opentelemetry/api';
import { setTimeout as sleep } from 'node:timers/promises';

const tracer = trace.getTracer('checkout-service');

async function processCheckout(userId) {
  return tracer.startActiveSpan('checkout', async (span) => {
    span.setAttribute('user.id', userId);

    await tracer.startActiveSpan('payment.charge', async (child) => {
      await sleep(120); // simulate payment call
      child.setAttribute('payment.provider', 'stripe');
      child.end();
    });

    span.end();
    return { ok: true };
  });
}

await processCheckout('user-42');
console.log('Checkout complete — check Jaeger at http://localhost:16686');
```

### Run to green

```bash
node --import ./src/instrumentation.js src/app.js
```

Expected output:

```
Checkout complete — check Jaeger at http://localhost:16686
```

Open http://localhost:16686, select `checkout-service` from the service dropdown, and click **Find Traces**. You should see one trace with two spans (`checkout` → `payment.charge`) and timing data. If the trace appears, the setup is complete. Move on.

## 2. Implement + best practice

### Structured logging — JSON over prose

```js
// ✓ structured: queryable in one command
log.info({ event: 'checkout.failed', user_id: userId, reason: err.code,
           trace_id: span.spanContext().traceId, duration_ms: elapsed });

// ✗ unstructured: grep archaeology on gigabytes
log.error('Checkout failed for user ' + userId + ' after ' + elapsed + 'ms');
```

The structured form enables `event="checkout.failed" AND user_id="42"` in Loki or Datadog in under a second. The unstructured form requires a regex scan across gigabytes of log files. Always include `trace_id` in every log line so that a log entry can be jumped to its parent trace in one click.

### Inject trace context into structured log lines

```js
import { context, trace } from '@opentelemetry/api';

function logWithTrace(fields) {
  const span = trace.getActiveSpan();
  const ctx = span ? span.spanContext() : null;
  console.log(JSON.stringify({
    ...fields,
    trace_id: ctx?.traceId,
    span_id:  ctx?.spanId,
  }));
}
```

This is the correlation ID pattern: every log line carries `trace_id` so an alert → log → trace investigation takes three clicks, not three hours.

### Metric cardinality — keep labels low

```js
// ✓ low cardinality: 3 values × 5 paths = 15 time series
const requestCounter = meter.createCounter('http.requests', {
  description: 'Requests by status class and path template',
});
requestCounter.add(1, {
  status_class: statusClass(res.statusCode), // '2xx' | '4xx' | '5xx'
  path: req.route?.path ?? 'unknown',        // '/api/checkout' not '/api/user/42/checkout'
});

// ✗ high cardinality: user_id has millions of values → millions of time series
requestCounter.add(1, { user_id: req.user.id, path: req.url });
```

Prometheus scales by time-series count. One label with a million distinct values creates a million time series. Move high-cardinality debugging to traces (`span.setAttribute('user.id', id)`) — spans are indexed differently and tolerate it.

### Span attributes vs events — when to use each

```js
tracer.startActiveSpan('payment.charge', (span) => {
  // Attribute: a property of the span (set once, describes the operation)
  span.setAttribute('payment.provider', 'stripe');
  span.setAttribute('payment.amount_cents', 4999);

  // Event: something that happened during the span (timestamped point)
  span.addEvent('retry_attempted', { attempt: 1, delay_ms: 500 });

  span.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
  span.end();
});
```

Attributes describe _what the span is_. Events describe _what happened while it ran_. Use `setStatus(ERROR)` on exceptions — this makes Jaeger show the span in red and lets you filter `status=ERROR` traces.

### SLO-driven alerting

```yaml
# prometheus rule — SLO burn rate alert (preferred over threshold alert)
groups:
  - name: checkout.slo
    rules:
      - alert: CheckoutSLOBurnRateHigh
        expr: |
          (
            rate(http_requests_total{status_class="5xx",path="/api/checkout"}[5m])
            /
            rate(http_requests_total{path="/api/checkout"}[5m])
          ) > 0.01
        for: 5m
        labels:
          severity: page
        annotations:
          summary: "Checkout error rate > 1% for 5 minutes — SLO budget burning fast"
```

Symptom alert ("error rate > 1%") fires on user pain, not on internal state. Prefer this over `CPU > 80%` — a CPU spike that does not affect users is noise; a 1% error rate is signal.

## 3. Common pitfalls

- **Unstructured logs.** Writing `log.error("Payment failed for user " + id)` instead of `log.error({event:"payment.failed", user_id: id, reason: err.code})` makes every investigation an archaeology exercise. Fix: mandate JSON-structured logging with consistent field names (`event`, `user_id`, `trace_id`, `duration_ms`) as a definition-of-done criterion. Add the `pino` or `winston` JSON formatter in one shared logger module so all services inherit it automatically.

- **Tagging metrics by user ID.** High-cardinality labels (user IDs, session IDs, email addresses) blow up the metrics backend. Prometheus scales by time-series count, not by sample count. Fix: use low-cardinality labels only; move high-cardinality debugging to traces or structured log queries. This happens because developers reach for the most specific identifier without knowing the cost model.

- **Alerting on internal thresholds, not user impact.** "CPU > 80%" fires constantly and is ignored; "SLO error budget exhausts in < 6 hours" fires rarely and is acted on. Fix: derive alerts from SLO burn rates, not from arbitrary resource thresholds. Page on symptoms users feel — errors, slow responses — not on internal state that may not correlate with user pain.

- **Head-based trace sampling discards the interesting traces.** 1% random sampling at 10k RPS keeps 100 traces per second — but the errored trace or the 5-second outlier has only a 1% chance of surviving. Fix: use tail-based sampling (keep the trace based on outcome — error, latency threshold — not at ingestion time). This happens because most OTel SDKs default to head-based sampling for simplicity.

- **No correlation IDs across services.** A request that touches five services produces five log streams with no shared key. Fix: propagate `trace_id` (or `request_id`) through every service call, in both logs and HTTP headers, via OTel context propagation (`propagation.inject(context.active(), headers)`). Without it, connecting a frontend error to the backend failure that caused it is guesswork.

- **PII and secrets in logs.** Full request bodies, auth tokens, or email addresses in log lines are an audit finding and a breach risk. Fix: use field-level redaction at the logging library level — pino's `redact` option, or a custom serialiser. This happens because developers log "the whole object" for debugging convenience and never remove it before shipping to production.

- **Forgetting source maps on deploy.** A minified production JS error (`at t.e (main.abc123.js:1:4820)`) is unreadable without the original source. Fix: upload source maps to your error tracker (Sentry, Datadog RUM) as part of every deploy step. Wire this into the CI/CD pipeline, not as a manual step.

## 4. Maintain

<div role="list" aria-label="Maintenance triggers and responses">
<Maintain trigger="OpenTelemetry SDK ships a new minor (e.g. 0.57 → 0.58)">
  1. Read the OTel JS changelog at https://github.com/open-telemetry/opentelemetry-js/releases for breaking API changes or deprecated exporters.
  2. Bump `@opentelemetry/sdk-node`, `@opentelemetry/auto-instrumentations-node`, and `@opentelemetry/exporter-trace-otlp-http` together — they must share the same semver minor.
  3. Run your app with `node --import ./src/instrumentation.js src/app.js` and confirm traces still land in Jaeger.
  4. Check for renamed span attributes — OTel semantic conventions update frequently and attribute names can change between minors.
  5. Update the `verified.versions` block and `verified.date` in frontmatter.
</Maintain>

<Maintain trigger="Traces stop appearing in Jaeger after a dependency update">
  1. Check the OTLP exporter URL — Jaeger 1.35+ uses port 4318 (HTTP) or 4317 (gRPC). Earlier versions used 14268. Confirm with `docker logs jaeger`.
  2. Check that `sdk.start()` is called before any `require`/`import` of instrumented libraries (http, express, etc.) — auto-instrumentation patches modules at load time.
  3. Add `OTEL_LOG_LEVEL=debug` to your environment to surface SDK-level errors: `OTEL_LOG_LEVEL=debug node --import ./src/instrumentation.js src/app.js`.
  4. Confirm the service name is set — traces without a `service.name` resource attribute are not queryable by name in Jaeger.
</Maintain>

<Maintain trigger="Alert fires on CPU/memory but no user impact is visible in traces">
  1. This is a signal that alerts are threshold-based rather than SLO-based. The firing alert is noise.
  2. Identify the user-facing SLO for the affected service (e.g. `checkout p99 < 500ms, error rate < 0.1%`).
  3. Replace or supplement the resource alert with a Prometheus burn-rate rule on the SLO metric.
  4. Silence the CPU/memory alert or raise its threshold until it only fires when correlated with a real SLO breach.
</Maintain>

<Maintain trigger="A log investigation is slow because log lines lack trace_id">
  1. Add the `trace_id` injection helper (Section 2) to every service that emits logs.
  2. Confirm that the structured log output contains `trace_id` and `span_id` fields by running the app locally and inspecting stdout.
  3. In your log backend (Loki, Datadog, Splunk), create a saved query that jumps from `trace_id` to the corresponding Jaeger trace in one click.
  4. Update the Definition of Done to require `trace_id` in all log lines as a merge criterion.
</Maintain>

<Maintain trigger="Metric cardinality is exploding — Prometheus memory is growing unbounded">
  1. Identify high-cardinality labels: `curl -s http://localhost:9090/api/v1/label/__name__/values | jq '.data | length'` shows total metric count; `topk(10, count by (__name__)({__name__=~".+"}))` shows the biggest series.
  2. Find the source label: any label with more than ~100 distinct values is a cardinality risk.
  3. Remove the offending label from the metric definition and move that dimension to a trace span attribute.
  4. Add a Prometheus recording rule or alert that fires when a single metric exceeds 10 000 time series as a future guard.
</Maintain>
