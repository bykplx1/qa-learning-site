## Brief

Build a production-grade k6 performance testing suite against
[QuickPizza](https://quickpizza.grafana.com) that covers three distinct load
patterns, enforces a centralised performance budget as a CI gate, and produces
a documented analysis of the system's behaviour under stress.

This is a portfolio artifact. It should read like something a senior
performance engineer would ship before a release — not a demo script.

> **Drift notice.** For reproducible CI runs, use the pinned Docker image
> (`grafana/quickpizza-local`) and point your scripts at
> `http://localhost:3333`. If you run against the live site at
> `https://quickpizza.grafana.com`, note this in your README and be aware that
> shared infrastructure may affect your results. Either approach is valid;
> document which you chose and why.

## Suggested approach

### Phase 1 — Load baseline (Day 1)

1. Set up a structured k6 project with a `src/` directory for scripts and a `config/` directory for shared options.
2. Create `config/budget.js` exporting your performance budget thresholds:
   ```js
   export const budget = {
     'http_req_failed': ['rate<0.01'],
     'http_req_duration': ['p(95)<500', 'p(99)<1000'],
   };
   ```
3. Write `src/load.js` — a `ramping-vus` scenario that ramps from 0 to 50 VUs over 5 minutes, holds for 10 minutes, then ramps down. Import and spread `budget` in `options.thresholds`.
4. Run against the Docker image. Document your baseline p(95) and error rate in `docs/perf-analysis.md`.

### Phase 2 — Stress and soak tests (Days 2–3)

5. Write `src/stress.js` — ramp to 2–3× your baseline VU count until thresholds breach. Document the VU count at which the system degraded and what metric crossed first.
6. Write `src/soak.js` — hold 20 VUs for 30 minutes. Look for memory leaks or p(99) drift over time. Add a custom trend metric (`new Trend('soak_duration')`) to track iteration duration across the soak window.
7. For each test type, add a section to `docs/perf-analysis.md` answering: *What failure mode did this reveal? What would you monitor in production to catch this earlier?*

### Phase 3 — CI performance budget gate (Days 3–4)

8. Add a `.github/workflows/perf-gate.yml` that:
   - Spins up the QuickPizza Docker container as a service
   - Runs `k6 run src/load.js` with `--out json=results.json`
   - Generates an HTML report using `k6 report results.json` (or `handleSummary` export)
   - Uploads the HTML report as a CI artifact
   - Exits non-zero and fails the job when any threshold is breached
9. Confirm that a deliberate threshold violation (temporarily tighten one threshold) causes the CI job to fail. Document this in your README.

### Phase 4 — Analysis and writeup (Day 5)

10. Finalise `docs/perf-analysis.md` with:
    - A baseline table (p(50), p(95), p(99), error rate) per load pattern
    - The stress test VU ceiling and what metric breached first
    - At least one threshold you relaxed after measuring and the data-driven justification
    - A recommendation: if this were a real service, what SLO would you propose and why?

## What "done" looks like

A public GitHub repo with:
- `src/load.js`, `src/stress.js`, `src/soak.js` all importing from `config/budget.js`
- A `.github/workflows/perf-gate.yml` that fails on threshold breach and uploads an HTML report
- Green CI badge confirmed at submission time
- `docs/perf-analysis.md` with baseline data, stress ceiling, soak findings, and SLO recommendation

## Why this matters

Load tests that only run locally are not performance tests — they're performance
experiments. The difference is enforcement: a CI gate that breaks the build
when a threshold regresses is the mechanism that prevents performance debt from
accumulating silently.

The three test types exist because load, stress, and soak each reveal different
failure modes: saturation, ceiling, and leak. Doing all three — and writing
the analysis — is the evidence that you can have the conversation with
engineering that precedes every SLO negotiation.
