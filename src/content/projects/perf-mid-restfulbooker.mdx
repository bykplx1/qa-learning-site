## Brief

Build a scenario-based k6 suite against
[restful-booker](https://restful-booker.herokuapp.com) that models two distinct
user workloads — read-heavy browsing and write-heavy booking creation —
enforces per-scenario thresholds, and runs in CI so threshold failures break
the build.

The lesson shows you how k6 scenarios and executors work. This project makes
you map real traffic shapes onto a booking domain: a guest browsing available
rooms has different latency expectations than a guest completing a reservation.

> **Drift notice.** For reproducible CI runs, use the local Docker image:
> `docker run -p 3001:3001 mwinteringham/restfulbooker` and point your
> `baseURL` at `http://localhost:3001`. Note in your README whether your CI
> pipeline runs against the live Heroku instance or the Docker image, and why.
> The Heroku free tier may spin down between requests — factor this into your
> threshold decisions.

## Suggested approach

1. Start from a clean k6 project or your starter script.
2. Obtain an auth token from `POST /auth` at the start of your setup function — restful-booker requires a token for mutating requests.
3. Define two scenarios in the `options.scenarios` object. Recommended pair:
   - `browse` — a `constant-vus` executor (5 VUs, 30s) hitting `GET /booking` and `GET /booking/:id` — simulates guests browsing reservation listings.
   - `create` — a `ramping-vus` executor (0→10→0 VUs over 1 min) hitting `POST /booking` — simulates a burst of reservation creation.
4. Tag each request with the scenario name so thresholds can be scoped:
   ```js
   http.get(`${baseURL}/booking`, { tags: { scenario: 'browse' } });
   ```
5. Define per-scenario thresholds:
   ```js
   'http_req_duration{scenario:browse}': ['p(95)<1500'],
   'http_req_duration{scenario:create}': ['p(99)<3000'],
   'http_req_failed{scenario:browse}': ['rate<0.05'],
   ```
6. Add a `.github/workflows/k6.yml` that:
   - Spins up the Docker container as a service (or points at the live API)
   - Runs `k6 run --summary-export=summary.json script.js`
   - Uploads `summary.json` as an artifact
   - Fails the job when k6 exits non-zero
7. Add the CI status badge to your README.

## What "done" looks like

A public GitHub repo with:
- A `script.js` with two named scenarios, per-scenario tags, and per-scenario thresholds
- A `.github/workflows/k6.yml` that fails on threshold breach and uploads `summary.json`
- A short README paragraph explaining one threshold decision — why that percentile and budget for that scenario
- A green CI badge in the README

## Why this matters

A booking API serves two very different audiences simultaneously: guests who
browse are tolerant of a 1.5-second response; guests who are mid-checkout are
not. Modelling these as separate scenarios with separate thresholds is the
difference between a test that gives you signal and a test that averages away
the detail you care about.
