## Brief

Build a production-grade k6 performance testing suite against a local
[JSON Server](https://github.com/typicode/json-server) instance backed by the
[JSONPlaceholder](https://jsonplaceholder.typicode.com) dataset — covering three
distinct load patterns, enforcing a centralised performance budget as a CI gate,
and producing a documented analysis of the system's behaviour.

Using a local JSON Server gives you full control over the target, eliminates
external network variability, and lets you observe the server's behaviour under
genuine stress without affecting shared infrastructure.

> **Drift notice.** This project uses JSON Server locally for CI reproducibility.
> Seed your server with a `db.json` file containing the JSONPlaceholder dataset
> (users, posts, comments, albums, photos, todos — freely downloadable from the
> JSONPlaceholder repo). The live `jsonplaceholder.typicode.com` endpoint is
> available for exploratory runs but should not be used as the CI target —
> external network latency will make your load, stress, and soak results
> non-reproducible. Document your seed data version and JSON Server version in
> your README.

## Suggested approach

### Phase 1 — Local target & load baseline (Day 1)

1. Set up a k6 project with `src/` for scripts and `config/` for shared options.
2. Add JSON Server as a dev dependency and create a `db.json` seed (or download
   it from the JSONPlaceholder GitHub repo). Add an npm script to start the server:
   `"server": "json-server --watch db.json --port 4000"`.
3. Create `config/budget.js` exporting your baseline thresholds:
   ```js
   export const budget = {
     'http_req_failed': ['rate<0.01'],
     'http_req_duration': ['p(95)<150', 'p(99)<300'],
   };
   ```
   Local JSON Server is fast — 150 ms p(95) is a realistic starting point.
4. Write `src/load.js` — a `ramping-vus` scenario that ramps from 0 to 30 VUs
   over 3 minutes, holds for 5 minutes, then ramps down. Cover both read
   (`GET /posts`, `GET /posts/:id`, `GET /posts/:id/comments`) and write
   (`POST /posts`, `PUT /posts/:id`) paths.
5. Run with `npm run server` in one terminal and `k6 run src/load.js` in another.
   Document your baseline p(95) and error rate in `docs/perf-analysis.md`.

### Phase 2 — Stress and soak tests (Days 2–3)

6. Write `src/stress.js` — ramp to 3–4× your baseline VU count. JSON Server is
   single-threaded (Node.js); it will degrade predictably as VUs exceed the event
   loop capacity. Document the VU count at which p(99) first exceeded your budget,
   and what the error rate looked like at the ceiling.
7. Write `src/soak.js` — hold 20 VUs for 20 minutes. Add a custom Trend metric
   (`new Trend('write_latency')`) tracking POST /posts iteration duration. JSON
   Server accumulates writes in memory; look for p(99) drift as the dataset grows.
8. For each test type, add a section to `docs/perf-analysis.md` answering:
   *What failure mode did this reveal? What would you monitor in production to
   catch this earlier?*

### Phase 3 — CI performance budget gate (Days 3–4)

9. Add a `.github/workflows/perf-gate.yml` that:
   - Installs dependencies and starts JSON Server as a background process
   - Waits for the server to be healthy (e.g. `curl --retry 5 --retry-delay 1 http://localhost:4000/posts`)
   - Runs `k6 run src/load.js` with `--out json=results.json`
   - Generates an HTML report using `handleSummary` or `k6 report`
   - Uploads the HTML report as a CI artifact
   - Exits non-zero and fails the job when any threshold is breached
10. Confirm the gate works: temporarily tighten one threshold to trigger failure.
    Document this in your README.

### Phase 4 — Analysis and writeup (Day 5)

11. Finalise `docs/perf-analysis.md` with:
    - A baseline table (p(50), p(95), p(99), error rate) per load pattern
    - The stress test VU ceiling and what metric breached first
    - Soak findings: did write latency drift as the dataset grew? Did error rate climb?
    - At least one threshold you relaxed after measuring, with data-driven justification
    - A recommendation: what would you change if this were a real multi-threaded
      API server instead of JSON Server, and why would that change your thresholds?

## What "done" looks like

A public GitHub repo with:
- `src/load.js`, `src/stress.js`, `src/soak.js` all importing from `config/budget.js`
- A `db.json` seed file (or a script that downloads it) and an npm start script for the server
- A `.github/workflows/perf-gate.yml` that starts the server, runs load tests, fails on threshold breach, and uploads an HTML report
- Green CI badge confirmed at submission time
- `docs/perf-analysis.md` with baseline data, stress ceiling, soak findings, and a comparison with what a production server would look like

## Why this matters

Running load tests against a controllable local target strips away the variable
you can't measure — external network latency — and isolates the variable you
care about: how your test design reveals system behaviour. Every finding in this
suite is reproducible by any engineer who clones the repo. That reproducibility
is what makes a performance analysis credible: not just "it was slow," but
"at 90 VUs, JSON Server's event loop saturated and p(99) hit 1.2 s — here is
the run log to prove it."
