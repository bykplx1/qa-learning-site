## Core Idea

Playwright becomes reliable only when a tester learns it as a small set of **orthogonal primitives**, not as Selenium-with-different-syntax. The load-bearing primitives are: **Locator** (lazy, retryable DOM handle), **auto-wait** (five conditions checked before every action), **web-first assertion** (retries until match or timeout), **fixture** (composable scoped setup/teardown), **browser context** (isolated storage inside one process), and **trace** (time-travel artefact of an entire run).

Flake is almost always a _primitive misuse_, not a Playwright bug. Race conditions, wrong assertion family, shared context state, missing auto-wait semantics — each maps to a primitive the tester hadn't internalised. The trace converts "this flakes sometimes" into "here is the millisecond it broke."

> Flake is a primitive-misuse diagnostic, not an environmental excuse. The trace is the evidence that makes diagnosis possible.

## 1. Set up

Everything from an empty folder to a single green Playwright run.

**Prerequisites:** Node ≥ 22.12.0, a terminal.

### Create the project

```bash
mkdir pw-runbook && cd pw-runbook
npm init -y
```

### Install Playwright (pinned)

```bash
npm install --save-dev @playwright/test@1.59.1
npx playwright install --with-deps chromium
```

> Pin the version. Playwright bundles its own Chromium — a bump regenerates visual baselines and can change trace timestamps. Only bump deliberately and regenerate baselines in the same PR.

### Scaffold the config

Create `playwright.config.ts` in the project root:

```ts
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests/e2e',
  fullyParallel: true,
  retries: process.env.CI ? 2 : 0,
  reporter: process.env.CI ? 'github' : 'list',
  use: {
    baseURL: 'https://example.com',
    trace: 'on-first-retry',
  },
});
```

### Write a first test

Create `tests/e2e/smoke.spec.ts`:

```ts
import { test, expect } from '@playwright/test';

test('home page loads', async ({ page }) => {
  await page.goto('/');
  await expect(page).toHaveTitle(/Example/);
});
```

### Run to green

```bash
npx playwright test
```

Expected output:

```
Running 1 test using 1 worker
  ✓  tests/e2e/smoke.spec.ts:3:5 › home page loads (1.2s)

  1 passed (2.5s)
```

If the run is green, the setup is complete. Move on.

## 2. Implement + best practice

### Locator hierarchy — use role first

```ts
// ✓ role-based: durable + a11y signal in one
const btn = page.getByRole('button', { name: 'Save' });

// ✓ label-based for form inputs
const email = page.getByLabel('Email address');

// ✗ last resort — CSS breaks on refactor and misses a11y
const btn = page.locator('.save-btn');
```

Follow this hierarchy: `getByRole` → `getByLabel` → `getByPlaceholder` → `getByText` → `getByTestId` → CSS → XPath. CSS and XPath are last resorts.

### Web-first assertions — always retry

```ts
// ✓ web-first: polls until the element is visible
await expect(gate).toBeVisible({ timeout: 10_000 });

// ✗ raw DOM read: reads once, fails silently before hydration
const text = await gate.textContent();
expect(text).toBeTruthy();
```

Use `expect(locator).toHaveText()` not `expect(await locator.textContent()).toBe()`. They look identical and behave oppositely under async UI.

### Fixtures for auth state

```ts
// fixtures/auth.ts
import { test as base } from '@playwright/test';

type AuthFixtures = { authedPage: Page };

export const test = base.extend<AuthFixtures>({
  authedPage: async ({ page }, use) => {
    // Hit your auth endpoint directly — no UI
    const res = await page.request.post('/api/auth/sign-in/social', {
      data: { provider: 'github', callbackURL: '/' },
      headers: { 'content-type': 'application/json' },
    });
    expect(res.ok()).toBeTruthy();
    await use(page);
  },
});
```

`page.request` shares the browser's cookie jar. The resulting session cookie is visible to subsequent `page.goto()` calls without a `storageState` file — correct for test-scope auth patterns.

For worker-scope (expensive) auth: write a `globalSetup` that logs in once and writes `storageState` to a file. Each test loads it via `use: { storageState: 'auth.json' }`.

### Intercepting overlays

```ts
// Register before the test body — fires when the overlay appears
await page.addLocatorHandler(page.locator('vite-error-overlay'), async (el) => {
  await el.locator('button').first().click();
});
```

Without this, an overlay covering your target element causes the next `click()` to timeout on the "receives events" auto-wait condition.

### Route interception — order matters

```ts
// ✓ register the listener BEFORE the action that triggers it
const [response] = await Promise.all([
  page.waitForResponse('**/api/save'),
  page.click(saveBtn),
]);
```

`page.route()` and `page.waitForResponse()` registered after the triggering action may miss the request entirely.

### Test step for readable traces

```ts
test('checkout flow', async ({ page }) => {
  await test.step('add item to cart', async () => {
    await page.click('[data-testid="add-to-cart"]');
    await expect(page.getByRole('status')).toHaveText('1 item');
  });

  await test.step('proceed to checkout', async () => {
    await page.getByRole('link', { name: 'Checkout' }).click();
    await expect(page).toHaveURL(/checkout/);
  });
});
```

`test.step()` names the action in the trace viewer timeline. Without it, a 20-action test is a flat list — with it, failures surface the step name immediately.

## 3. Common pitfalls

- **Reaching for `getByTestId` or CSS before trying `getByRole`.** Role-based locators are more durable (survive text/class refactors) and double as an a11y assertion. Fix: follow the hierarchy — `getByRole` first, then `getByLabel`, `getByPlaceholder`, `getByText`, finally `getByTestId`. CSS and XPath are last resorts. This happens because CSS and XPath are familiar from other toolchains and feel concrete.

- **Using `expect(await locator.textContent()).toBe('Foo')` on async UI.** This reads the DOM once — if the value hasn't updated yet, it fails silently. The form `await expect(locator).toHaveText('Foo')` retries until match or timeout. These two look nearly identical and behave oppositely under any UI that updates after first paint. Fix: use the web-first assertion form for any state that can change after navigation or action.

- **`page.waitForTimeout(ms)` to stabilise a flaky step.** Sleep-based waits silently break when underlying timing changes — they increase wall-clock without improving reliability. Fix: replace with the appropriate `expect(locator).toBeVisible()` or `expect(locator).toHaveAttribute(...)` assertion and let auto-wait do the work.

- **Reusing captured `storageState` across tests that mutate auth state.** `globalSetup` logs in once and writes cookies to a file; subsequent tests re-hydrate from that file. A test that logs out or deletes the account invalidates the file for every remaining test in the worker. Fix: use per-test isolated accounts for flows that mutate session state; use worker-scope fixtures only for immutable shared resources.

- **Registering `page.route()` or `page.waitForResponse()` after the action that triggers them.** Routes registered after a navigation don't intercept the request; `waitForResponse` after a click may miss a response that arrived first. Fix: wrap both in `Promise.all` so the listener is registered before the action.

- **Leaving `page.pause()` in committed tests.** A stray `pause()` blocks CI workers indefinitely; the symptom is "the suite is hanging" with no timeout error. Fix: add a lint rule or CI grep that fails on committed `pause()` calls.

- **Committing visual baselines from a non-Linux machine.** Font rendering, antialiasing, and GPU rasterisation differ across macOS/Linux/Windows even with Playwright's bundled Chromium. Baselines regenerated on one OS will fail on another. Fix: regenerate baselines in CI via `workflow_dispatch` with `update_visual_baselines=true`. Treat the pinned `@playwright/test` version as part of the baseline — bumping the version requires regenerating baselines in the same PR.

## 4. Maintain

<div role="list" aria-label="Maintenance triggers and responses">
<Maintain trigger="Playwright ships a major version (e.g. 1.x → 2.x)">
  1. Read the changelog for breaking API changes.
  2. Bump `@playwright/test` in `package.json` to the new pinned version.
  3. Run `npx playwright install --with-deps chromium` to update the browser bundle.
  4. Run the full test suite — most failures will be timeout changes or renamed options.
  5. Regenerate visual baselines in CI (`workflow_dispatch update_visual_baselines=true`). Do not commit baselines from your local machine.
  6. Update the `verified.versions` block and `verified.date` in frontmatter.
</Maintain>

<Maintain trigger="A locator breaks after a UI refactor (element not found)">
  1. Run the failing spec with `--trace on` to get a trace file.
  2. Open the trace: `npx playwright show-trace trace.zip`. Check the DOM snapshot at the failing action.
  3. Find the new accessible name or role using browser DevTools accessibility panel.
  4. Update the locator — prefer moving *up* the hierarchy (e.g. CSS → `getByRole`) rather than patching the CSS selector.
  5. If the element genuinely has no role, add `data-testid` to the markup and use `getByTestId`.
</Maintain>

<Maintain trigger="A test flakes 1-in-N runs in CI">
  1. Enable trace on retry: set `trace: 'on-first-retry'` in `playwright.config.ts`.
  2. Reproduce in CI or run locally with `--repeat-each=10`.
  3. Open the trace of the failed run. Look for: the exact millisecond the action fired, whether the element was attached/visible/enabled at that moment, network requests that were still in-flight.
  4. Common fixes: replace `waitForTimeout` with a web-first assertion, move `page.route()` registration before the action that triggers it, switch from worker-scope to test-scope fixture if the fixture state is mutated.
  5. Never add `waitForTimeout` as a "fix" — it masks the root cause and makes the suite slower.
</Maintain>

<Maintain trigger="Visual baselines are consistently failing in CI (screenshot diff > threshold)">
  1. Check whether a Playwright version bump occurred — this always invalidates baselines.
  2. Check whether the diff is content-driven (a genuine UI change) or rendering-driven (font/antialiasing drift).
  3. For content changes: update the relevant component, then regenerate baselines via `workflow_dispatch`.
  4. For version bumps: regenerate all baselines in CI in the same PR as the version bump.
  5. Never regenerate baselines from a local (non-Linux) machine — cross-OS rendering differs.
</Maintain>

<Maintain trigger="Auth state fixture stops working (401 on previously-authed routes)">
  1. Check whether the auth provider changed its callback/token shape (common on OAuth provider updates).
  2. If using `storageState` files: delete `auth.json` and regenerate via `globalSetup`.
  3. If using `page.request` inline: inspect the response body — the sign-in endpoint may now return a different redirect URL structure.
  4. For test-scope auth: the issue is usually a test that mutates session state (sign-out, account deletion) — isolate those tests into their own browser context.
</Maintain>
