## Core Idea

Performance testing is the practice of asking "how does this system behave under realistic and adversarial workloads?" — *before* production answers the question for you. It is not a single test type; it is a family of **workload shapes** run against an SLO-defined target, with results read as **distributions**, not averages.

The load-bearing claims: a performance number without a **percentile** and a **workload shape** is meaningless. "Our API responds in 80 ms" is not a result. "Our `GET /search` p95 is 240 ms at 500 RPS sustained for 30 minutes, cache warm" is a result. The [[test-oracles-and-prioritization]] principle applies directly — without an SLO defining pass/fail, the perf test has no verdict.

Performance testing also spans two distinct disciplines: **backend perf** (server response distributions under concurrency) and **frontend perf** (user-experienced paint metrics). Both use the word "performance"; they answer different questions.

> A performance number without a percentile and a workload shape is a feeling, not a measurement. The QA contribution is converting "the site feels slow" into "p95 LCP is 3.2 s at p75 of real users — fail by 28%."

## Worked Example

### Backend perf: from vibes to verdict

A team reports "the checkout API is slow." Start by defining the oracle — no oracle, no test:

```
SLO (agreed with team before writing one line of test code):
  POST /checkout
  p95 latency ≤ 400 ms
  p99 latency ≤ 800 ms
  Error rate ≤ 0.1%
  At: 300 RPS sustained for 15 minutes, cache warm, production-shaped data (10M orders)
```

Next, the k6 load test matching that SLO:

```js
// k6/checkout-load.js
import http from 'k6/http';
import { sleep } from 'k6';

export const options = {
  // Open-model executor: requests arrive at a fixed rate regardless of response time
  // This avoids coordinated omission — slow responses do not throttle the test
  scenarios: {
    load: {
      executor: 'constant-arrival-rate',
      rate: 300,          // 300 requests per second
      timeUnit: '1s',
      duration: '15m',
      preAllocatedVUs: 50,
      maxVUs: 200,
    },
  },
  thresholds: {
    http_req_duration: ['p(95)<400', 'p(99)<800'],  // fail if SLO breached
    http_req_failed: ['rate<0.001'],                 // fail if error rate > 0.1%
  },
};

export default function () {
  const payload = JSON.stringify({ productId: Math.floor(Math.random() * 1000) + 1 });
  http.post('https://staging.example.com/checkout', payload, {
    headers: { 'Content-Type': 'application/json' },
  });
  sleep(2); // model real user think-time — not a delay, a workload shape decision
}
```

**Two decisions in that script that most teams get wrong:**

1. `constant-arrival-rate` instead of `constant-vus`: the closed model (`constant-vus`) waits for a response before sending the next request. If a response takes 5 s, no request is sent for 5 s — the test *under-counts* slow responses. This is **coordinated omission**. The open model (`constant-arrival-rate`) sends requests on schedule regardless, exposing the true tail.

2. `sleep(2)` between iterations: real users read a confirmation page before clicking again. Without think-time, 50 VUs generate 50× the real concurrency. The sleep is not test overhead — it is workload modelling.

### Frontend perf: Lighthouse-CI as a shift-left gate

The [[cicd-for-testing]] topic already shows this site's `lhci` job. Here is the budget it enforces and what each metric means for the user:

```json
// lighthouserc.json (simplified)
{
  "ci": {
    "assert": {
      "assertions": {
        "largest-contentful-paint": ["error", { "maxNumericValue": 2500 }],
        "interaction-to-next-paint": ["error", { "maxNumericValue": 200 }],
        "cumulative-layout-shift": ["error", { "maxNumericValue": 0.1 }],
        "total-blocking-time": ["warn", { "maxNumericValue": 300 }]
      }
    }
  }
}
```

- **LCP ≤ 2.5 s** — the largest element on screen has painted. Users perceive this as "the page loaded."
- **INP ≤ 200 ms** — Interaction to Next Paint (replaced FID in March 2024). Samples *all* interactions per user, reports the worst tail. A site passing FID often fails INP because INP is harsher.
- **CLS ≤ 0.1** — layout stability. A score of 0 means nothing jumped; 0.25+ means elements are shifting after paint.

Budget the *individual metrics*, not the composite Lighthouse score. The score is a weighted blend — a regression in LCP can hide behind an improvement in SEO. The score tells you "something changed"; the metrics tell you what.

### Fan-out tail latency — the math teams ignore

A page calls 5 backend services in parallel. Each has p99 = 200 ms. What is the page's p99?

Not 200 ms. The page completes when the *slowest* of the 5 responses returns. If each independently has a 1% chance of taking 200+ ms, the probability that *at least one* of the five takes 200+ ms is:

```
P(at least one slow) = 1 - P(all fast)
                     = 1 - (0.99)^5
                     ≈ 4.9%
```

So the page's p99 is actually the individual service's ~p95. With 10 parallel services it approaches the individual p90. This is **fan-out amplification** — "fast services" do not automatically produce fast pages. The lesson: instrument the *page* p99, not just the service p99. A [[playwright]] trace that shows 5 parallel fetches is a signal worth measuring under load.

## Common Pitfalls

- **Reporting average response time.** Averages hide the tail. 990 requests at 50 ms and 10 requests at 5 s average to ~100 ms — which looks fine and is not. Fix: always report p50, p95, p99 (and p99.9 when the SLO demands). Ban the word "average" except for explicitly-named throughput metrics. This happens because most dashboards default to mean; change the default.

- **Testing against a tiny database.** A perf test against a 1,000-row table reveals nothing about a 10-million-row production table. Query plans flip on data volume; indexes that work at small scale stop working at scale. Fix: seed production-shaped data (or a production clone with PII scrubbed) before running perf tests. Treat data setup as part of the test, not optional overhead.

- **Coordinated omission with a closed-model executor.** `constant-vus` (the default in many tools) waits for a response before sending the next request. Under saturation, slow responses throttle the test — which then *under-reports* tail latency. Fix: use `constant-arrival-rate` (k6) or `injectOpen` (Gatling) for tail metrics; reserve closed-model for "how does the system behave with exactly N concurrent users" questions. Name the executor in every result.

- **No SLO before writing the test.** A perf test without a pass/fail oracle is exploratory, not a quality gate. "Looks fine" is not a result. Fix: define the SLO with the team — percentile, threshold, workload shape, cache state, data state — before writing one line of test code. The SLO is the oracle (back-reference: [[test-oracles-and-prioritization]]).

- **Budgeting the Lighthouse composite score instead of individual metrics.** The score is a weighted blend; regressions in LCP can be masked by improvements elsewhere. Fix: assert on individual Core Web Vitals — LCP, INP, CLS — in `lighthouserc.json`. If only the score is budgeted, the team optimises the score, not the user experience.

- **Running perf tests with cold caches and calling the result "production performance."** Most production requests hit warm caches; a cold-cache result is a worst-case, not a typical case. Fix: document the cache state in every perf result. Run separate cold-start and warm-state tests; label them correctly.

- **Forgetting the warm-up period.** JIT compilation (JVM), connection-pool fill, and cache warming produce transient latency spikes in the first 5–10% of a test run. Including that data in steady-state results inflates the reported p99. Fix: add a ramp-up stage and discard results until the system stabilises; or build the warmup into the scenario and exclude warmup results from the threshold evaluation.

- **Reporting a single perf run as "the number."** Perf is a distribution; one run is one sample of that distribution. On shared CI runners, runner noise alone can account for 10–20% variance. Fix: run 3–5 times and report the median; flag outliers. When comparing across PRs on shared runners, use relative thresholds (vs. a same-job baseline) rather than absolute values. The [[cicd-for-testing]] pipeline's Lighthouse job is the right model for how to wire this.

## Practice Task

<PracticeTask id="perf-task-1" rubric="perf-rubric-v1">
  Define and defend a performance budget for one critical user flow on a site you control (or the QA learning site at `https://qa-learning-site.vercel.app`). The flow: **load the homepage → click a lesson → render the lesson page.**

  **Deliverable 1 — SLOs in writing**

  Define the following, with a specific number for each:
  - Frontend: LCP, INP, CLS threshold for the lesson page.
  - Backend: p95 latency threshold for any API the flow hits (e.g., progress, auth session check).
  - State the workload shape (which of the four canonical shapes), the cache state (cold/warm), and the data state (how many users/lessons seeded).

  **Deliverable 2 — Budget rationale (≤ 300 words)**

  For each SLO number, name *why this number* — cite at least one external anchor: CrUX p75 bands ("good" LCP ≤ 2.5 s), Google's INP thresholds (≤ 200 ms good, ≤ 500 ms needs improvement), a competitor benchmark, or historical data. A number without an anchor is a guess.

  **Deliverable 3 — Test plan**

  State which tests run where:
  - Per-PR gate: what tool, what metric, which stage of [[cicd-for-testing]] pipeline.
  - Pre-release: what tool, which workload shape, what environment.
  - Production validation: what RUM signal would you instrument, and with what alert threshold.

  **Deliverable 4 — Failure-mode plan**

  Define:
  - What triggers a hard build block (vs. a warning).
  - How you distinguish a real regression from noise-floor variance. Name a specific threshold (e.g., >10% increase over 3-run median).
  - What escalation path a breached SLO follows if the regression is confirmed.

  **Deliverable 5 — Evidence**

  Run Lighthouse-CI on the target site once (locally or in CI) and report the actual numbers vs. your proposed budget. If the numbers breach your budget, name the specific Lighthouse diagnostic you would read first and what it would tell you.

  Rubric (revealed after submission):
  - Does each SLO have a percentile AND a workload shape AND a cache state? Missing any one is a partial credit only.
  - Does the rationale cite an external anchor (CrUX, Google threshold, benchmark) rather than an internally-generated number? Unsupported numbers fail.
  - Does the test plan name a specific executor type for the backend load test (closed vs. open) and justify the choice?
  - Does the failure-mode plan define a noise-floor threshold (not just "we'll decide")? A threshold-free plan is a fail.
  - Does the evidence section report actual Lighthouse numbers, not "I would run Lighthouse"? Hand-waving fails.
  - Bonus: does the candidate name a failure mode the chosen workload shape will *not* catch, and name which alternative shape would catch it?
</PracticeTask>
