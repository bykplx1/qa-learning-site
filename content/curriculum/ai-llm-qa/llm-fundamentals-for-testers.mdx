## Core Idea

An LLM is **a function that takes a sequence of tokens and returns a probability distribution over the next token**, sampled to produce text. Every property a tester must reason about — non-determinism, context limits, cost, latency, hallucination, prompt sensitivity — falls out of that one sentence.

The five substrate facts that every LLM tester must hold: tokens (not words) drive cost and context limits; each output token is sampled from a probability distribution conditioned on all prior tokens; `temperature`, `top_p`, and `top_k` are *test variables*, not knobs; the context window is a hard ceiling with silent-truncation failure modes; and non-determinism is *structural* — `temperature=0` is greedy decoding, not determinism.

The load-bearing claim: **an LLM is a probability machine wrapped in a deterministic API; tests must respect both layers.** Treating it as a deterministic API produces flaky tests. Treating it as pure chaos produces no tests at all. The discipline is *bounded stochasticity*: assert on schemas, invariants, and distributions — never on exact strings. The oracle problem from [[test-oracles-and-prioritization]] is now the central problem of every test you write.

> The prompt is the program. The model is a stochastic runtime. Write tests for the program, not the runtime.

## Worked Example

### Testing a non-deterministic summary endpoint

A team has an LLM endpoint that summarises a support ticket into a one-line incident title. The first test a developer writes:

```ts
// BAD — will eventually fail even at temperature=0
it('summarises the ticket correctly', async () => {
  const result = await summarise('Login fails with error 500 on Safari 17');
  expect(result).toBe('Safari login error 500');
});
```

The mistake: `expect(result).toBe(...)` is an exact-string assertion against a stochastic system. It passes today. It fails when the model rounds a new deployment, when `temperature` is bumped from 0 to 0.3 for a quality improvement, or simply due to GPU-batch variance at temperature=0.

The fix: assert on invariants and schema, not on the literal string.

```ts
// GOOD — asserts on invariants that any valid output must satisfy
it('summarises the ticket into a valid title', async () => {
  const result = await summarise('Login fails with error 500 on Safari 17');

  // structural: the output is a non-empty string
  expect(typeof result).toBe('string');
  expect(result.length).toBeGreaterThan(0);

  // length invariant: a "title" should not be a paragraph
  expect(result.split(' ').length).toBeLessThanOrEqual(12);

  // content invariant: must reference the core failure concept
  // use a semantic check, not exact-string matching
  const lower = result.toLowerCase();
  expect(
    lower.includes('login') || lower.includes('auth') || lower.includes('safari')
  ).toBe(true);
});
```

For higher-stakes assertions — "does this summary accurately reflect the ticket?" — the invariant approach runs out and a rubric or LLM-as-judge is needed. But the schema + invariant layer catches the obvious regressions cheaply and deterministically.

**Log the assembled prompt at the API boundary.** The most common LLM debugging gap is the diff between what the developer *thinks* the prompt is and what *actually* hits the model. Without logging, regressions from prompt-template changes are invisible.

```ts
// Always log what was sent: model ID, sampling params, assembled prompt
const response = await client.messages.create({
  model: 'claude-opus-4-7',  // pin a dated ID, not an alias
  max_tokens: 60,
  temperature: 0.3,
  messages: [{ role: 'user', content: assembledPrompt }],
});
console.log({ model: response.model, usage: response.usage, prompt: assembledPrompt });
```

## Common Pitfalls

- **Asserting on exact string output.** → Assert on schema, invariants ("length ≤ 12 words", "contains a date in YYYY-MM-DD format"), or a rubric. → Because LLM output varies even at `temperature=0` due to GPU batch effects; exact-string tests are guaranteed to be flaky.

- **Estimating cost by word count or character count.** → Use the provider's own tokeniser (`tiktoken` for OpenAI; Anthropic's token-counting endpoint). → Because tokens diverge from words by 30–300%; code and non-English text can cost 3–10× more than naive character estimates suggest. A production cost model based on character counts can be off by an order of magnitude.

- **Treating `temperature=0` as deterministic.** → Run N≥5 samples even at temp=0; assert on distribution properties rather than a single run. → Because GPU batch composition changes which floating-point reductions execute; small numerical differences flip the top-probability token on ~0.1–1% of positions. OpenAI and Anthropic both acknowledge this. The fix is not retry-until-pass — that converts a real signal into noise.

- **Not pinning the model version.** → Pin to a dated model ID (e.g. `gpt-4o-2024-08-06`, `claude-opus-4-7`); log `system_fingerprint` or the provider's equivalent build identifier. → Because aliases like `gpt-4o` or `claude-3-5-sonnet` resolve to whichever build the provider currently points at. Same alias, different build = behaviour can change without any code change. Pinning is the only way to make results reproducible.

- **Ignoring context-window overflow silently.** → Token-count the assembled prompt before sending; fail loudly (not silently) on overflow. → Because LangChain et al. silently truncate the prompt — often from the start, deleting the system prompt without warning. This is the single most common "why did the model suddenly stop following instructions?" bug, and it has no stack trace.

- **Mixing `temperature` and `top_p` without controlling for the other.** → Vary one sampling parameter at a time in experiments; document the other's value. → Because `top_p=0.1` already constrains the distribution so tightly that temperature has little remaining effect; reporting "we tested at temperature=0.7" while leaving `top_p=0.1` fixed is not the experiment the team thinks it ran.

- **Testing the model rather than the system.** → Test the assembled prompt + model + post-processing as a unit; the model alone is a vendor artefact. → Because the prompt is *your code* and the model is a *dependency*. Tests bound the system's behaviour; they do not audit the vendor's model. "The model is bad at X" and "our system prompt steers the model away from X" are different diagnoses with different fixes, and only tests on the *assembled system* can tell them apart.
