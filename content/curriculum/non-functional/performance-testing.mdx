## Core Idea

Performance testing is the practice of asking "how does this system behave under realistic and adversarial workloads?" — *before* production answers the question for you. It is not a single test type; it is a family of **workload shapes** run against an SLO-defined target, with results read as **distributions**, not averages. The load-bearing claims: a performance number without a **percentile** and a **workload shape** is meaningless. "Our API responds in 80 ms" is not a result. "Our `GET /search` p95 is 240 ms at 500 RPS sustained for 30 minutes, cache warm" is a result. The [[test-oracles-and-prioritization]] principle applies directly — without an SLO defining pass/fail, the perf test has no verdict.

## 1. Set up

Everything from an empty folder to a single green k6 run.

**Prerequisites:** k6 v2.0.0, Node ≥ 22.12.0, a terminal.

### Install k6

k6 is a standalone binary — install it via your OS package manager or directly.

**macOS (Homebrew):**

```bash
brew install k6
```

**Linux (Debian/Ubuntu):**

```bash
sudo gpg -k
sudo gpg --no-default-keyring \
  --keyring /usr/share/keyrings/k6-archive-keyring.gpg \
  --keyserver hkp://keyserver.ubuntu.com:80 \
  --recv-keys C5AD17C747E3415A3642D57D77C6C491D6AC1D69
echo "deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] \
  https://dl.k6.io/deb stable main" \
  | sudo tee /etc/apt/sources.list.d/k6.list
sudo apt-get update && sudo apt-get install k6
```

**Windows (Chocolatey):**

```bash
choco install k6
```

**Verify the version:**

```bash
k6 version
# k6 v2.0.0 (...)
```

> Pin the version in CI. k6's built-in metrics and options change across releases; a bump can silently change threshold semantics.

### Create the project

```bash
mkdir k6-runbook && cd k6-runbook
```

No `npm init` needed — k6 scripts are plain JS/TS files executed by the k6 runtime, not Node.

### Write a smoke test

Create `smoke.js`:

```js
// smoke.js  — k6 2.0.0
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  vus: 1,         // 1 virtual user
  duration: '10s',
  thresholds: {
    http_req_failed: ['rate<0.01'],          // error rate < 1%
    http_req_duration: ['p(95)<1000'],       // p95 < 1 s
  },
};

export default function () {
  const res = http.get('https://httpbin.test.k6.io/get');
  check(res, { 'status 200': (r) => r.status === 200 });
  sleep(1);
}
```

### Run to green

k6 run smoke.js

## 2. Implement + best practice

### Define the SLO first — no oracle, no test

Write the SLO as a comment at the top of every script before writing one line of test code:

```js
// k6/checkout-load.js  — k6 2.0.0
//
// SLO (agreed with team):
//   POST /checkout
//   p95 latency ≤ 400 ms
//   p99 latency ≤ 800 ms
//   error rate ≤ 0.1%
//   at: 300 RPS sustained for 15 min, cache warm, 10 M order rows
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  scenarios: {
    load: {
      // Open model: requests arrive on schedule regardless of response time.
      // Avoids coordinated omission — slow responses do not throttle the test.
      executor: 'constant-arrival-rate',
      rate: 300,
      timeUnit: '1s',
      duration: '15m',
      preAllocatedVUs: 50,
      maxVUs: 200,
    },
  },
  thresholds: {
    http_req_duration: ['p(95)<400', 'p(99)<800'],
    http_req_failed:   ['rate<0.001'],
  },
};

export default function () {
  const payload = JSON.stringify({
    productId: Math.floor(Math.random() * 1000) + 1,
  });
  const res = http.post(
    'https://staging.example.com/checkout',
    payload,
    { headers: { 'Content-Type': 'application/json' } }
  );
  check(res, { 'status 200': (r) => r.status === 200 });
  sleep(2); // model real user think-time — not overhead, a workload shape decision
}
```

Two decisions that most teams get wrong:

1. `constant-arrival-rate` instead of `constant-vus`: the closed model waits for a response before sending the next. Under saturation, slow responses throttle the test — which then under-reports tail latency. This is **coordinated omission**. The open model sends on schedule regardless, exposing the true tail.

2. `sleep(2)` between iterations: real users read a confirmation page before clicking again. Without think-time, 50 VUs generate 50× the real concurrency. The sleep is workload modelling, not test overhead.

### Multi-scenario script — four shapes in one file

```js
// k6/all-shapes.js  — k6 2.0.0
import http from 'k6/http';
import { sleep } from 'k6';

export const options = {
  scenarios: {
    load: {
      executor: 'constant-arrival-rate',
      rate: 300, timeUnit: '1s', duration: '15m',
      preAllocatedVUs: 50, maxVUs: 200,
      startTime: '0s',
    },
    soak: {
      executor: 'constant-vus',
      vus: 200, duration: '8h',
      startTime: '16m', // run after load scenario
    },
  },
  thresholds: {
    'http_req_duration{scenario:load}': ['p(95)<400'],
    'http_req_duration{scenario:soak}': ['p(95)<600'],
  },
};

export default function () {
  http.get('https://staging.example.com/api/products');
  sleep(1);
}
```

Tag thresholds with `{scenario:name}` to assert different budgets per shape. Without the tag, thresholds blend both scenarios' data.

### Frontend perf: Lighthouse-CI as a shift-left gate

The [[cicd-for-testing]] topic shows this site's `lhci` job. The budget it enforces:

```json
{
  "ci": {
    "assert": {
      "assertions": {
        "largest-contentful-paint": ["error", { "maxNumericValue": 2500 }],
        "interaction-to-next-paint": ["error", { "maxNumericValue": 200 }],
        "cumulative-layout-shift": ["error", { "maxNumericValue": 0.1 }],
        "total-blocking-time": ["warn", { "maxNumericValue": 300 }]
      }
    }
  }
}
```

Budget the individual metrics, not the composite score. The score is a weighted blend — a regression in LCP can hide behind an improvement in SEO.

### Fan-out tail latency — the maths teams ignore

A page calls 5 backend services in parallel. Each has p99 = 200 ms. The page completes when the *slowest* of the 5 returns:

```
P(at least one slow) = 1 - P(all fast)
                     = 1 - (0.99)^5
                     ≈ 4.9%
```

So the page's p99 is actually the individual service's ~p95. With 10 parallel services it approaches the individual p90. This is **fan-out amplification**. Instrument the *page* p99, not just the service p99.

## 3. Common pitfalls

- **Reporting average response time.** Averages hide the tail. 990 requests at 50 ms and 10 requests at 5 s average to ~100 ms — which looks fine and is not. Fix: always report p50, p95, p99. Ban the word "average" except for explicitly-named throughput metrics. This happens because most dashboards default to mean; change the default.

- **Testing against a tiny database.** A perf test against a 1,000-row table reveals nothing about a 10-million-row production table. Query plans flip on data volume; indexes that work at small scale stop working at scale. Fix: seed production-shaped data (or a scrubbed production clone) before running perf tests. Treat data setup as part of the test, not optional overhead.

- **Coordinated omission with a closed-model executor.** `constant-vus` (the default) waits for a response before sending the next. Under saturation, slow responses throttle the test and under-report tail latency. Fix: use `constant-arrival-rate` for tail metrics; reserve `constant-vus` for "how does the system behave with exactly N concurrent users" questions. Name the executor in every result.

- **No SLO before writing the test.** A perf test without a pass/fail oracle is exploratory, not a quality gate. "Looks fine" is not a result. Fix: define the SLO with the team — percentile, threshold, workload shape, cache state, data state — before writing one line of test code. The SLO is the oracle (back-reference: [[test-oracles-and-prioritization]]).

- **Budgeting the Lighthouse composite score instead of individual metrics.** The score is a weighted blend; regressions in LCP can be masked by improvements elsewhere. Fix: assert on individual Core Web Vitals — LCP, INP, CLS — in `lighthouserc.json`. If only the score is budgeted, the team optimises the score, not the user experience.

- **Running perf tests with cold caches and calling the result "production performance."** Most production requests hit warm caches; a cold-cache result is a worst-case, not a typical case. Fix: document the cache state in every perf result. Run separate cold-start and warm-state tests; label them correctly.

- **Forgetting the warm-up period.** JIT compilation, connection-pool fill, and cache warming produce transient latency spikes in the first 5–10% of a test run. Including that data inflates the reported p99. Fix: add a ramp-up stage and discard results until the system stabilises; or build the warmup into the scenario and exclude warmup results from threshold evaluation.

- **Reporting a single perf run as "the number."** Perf is a distribution; one run is one sample of that distribution. On shared CI runners, runner noise alone can account for 10–20% variance. Fix: run 3–5 times and report the median; flag outliers. Use relative thresholds (vs. a same-job baseline) rather than absolute values.

## 4. Maintain

<div role="list" aria-label="Maintenance triggers and responses">
<Maintain trigger="k6 ships a new major version (e.g. 2.x → 3.x)">
  1. Read the k6 migration guide — executor option names and metric labels sometimes change between majors.
  2. Update the `k6` version in your CI install step and any pinned Docker images.
  3. Run the full perf test suite in a staging environment; compare p95/p99 results against the previous baseline to confirm no silent threshold-semantics change.
  4. Update the `verified.versions` block and `verified.date` in this frontmatter.
</Maintain>

<Maintain trigger="A threshold starts failing after a schema or infrastructure change (not a real regression)">
  1. Check whether the data volume, cache state, or connection-pool configuration changed — these affect latency independently of code changes.
  2. Re-run the test 3 times and compare the median with the previous 3-run median. If the delta is within the runner noise floor (≤10%), treat it as noise, not regression.
  3. If the delta is confirmed, bisect: run the test against the old build in the same environment. If the old build also breaches, the infra changed, not the code.
  4. Update the SLO threshold only after aligning with the team — do not silently bump numbers to make CI green.
</Maintain>

<Maintain trigger="A soak test surfaces a memory-leak or pool-exhaustion bug">
  1. Note the time at which the metric diverged (p95 climbing, error rate rising) — this anchors the debugging window.
  2. Correlate with application metrics: heap usage, DB connection count, active thread count. The divergence point in k6 should match a corresponding inflection in the application metric.
  3. Add a `check` in the k6 script to assert response body structure, not just status code — pool exhaustion often returns 200 with an empty or error body.
  4. Fix the resource leak, then run a new soak test for the same duration to confirm the metric is flat over time, not just low at the start.
</Maintain>

<Maintain trigger="CI perf job runtime exceeds the pipeline time budget">
  1. Move soak tests (8+ hours) out of per-PR CI and into a scheduled nightly or pre-release gate.
  2. Run only the smoke and load shapes per PR; reserve stress and soak for the release gate.
  3. Use k6 Cloud or a dedicated runner for long-running scenarios to avoid competing with unit/integration jobs for resources.
  4. Document which shapes run where in the team runbook so a future engineer does not re-add a soak test to the PR gate.
</Maintain>

<Maintain trigger="Lighthouse-CI budgets are consistently failing after a UI change">
  1. Run Lighthouse locally against the production build: `npx lhci autorun`.
  2. Identify which Core Web Vital is failing — LCP, INP, or CLS — not the composite score.
  3. For LCP: check whether a new image or font was added without preload. For CLS: check for newly-added `client:only` islands that shift layout on mount. For INP: profile with Chrome DevTools Performance panel — look for long tasks blocking the main thread.
  4. Fix the root cause, then re-run Lighthouse 3 times and report the median value vs. the budget before updating the CI config.
</Maintain>
</div>

## Practice Task

<PracticeTask id="perf-task-1" rubric="perf-rubric-v1">
  Define and defend a performance budget for one critical user flow on a site you control (or the QA learning site at `https://qa-learning-site.vercel.app`). The flow: **load the homepage → click a lesson → render the lesson page.**

  **Deliverable 1 — SLOs in writing**

  Define the following, with a specific number for each:
  - Frontend: LCP, INP, CLS threshold for the lesson page.
  - Backend: p95 latency threshold for any API the flow hits (e.g., progress, auth session check).
  - State the workload shape (which of the four canonical shapes), the cache state (cold/warm), and the data state (how many users/lessons seeded).

  **Deliverable 2 — Budget rationale (≤ 300 words)**

  For each SLO number, name *why this number* — cite at least one external anchor: CrUX p75 bands ("good" LCP ≤ 2.5 s), Google's INP thresholds (≤ 200 ms good, ≤ 500 ms needs improvement), a competitor benchmark, or historical data. A number without an anchor is a guess.

  **Deliverable 3 — Test plan**

  State which tests run where:
  - Per-PR gate: what tool, what metric, which stage of [[cicd-for-testing]] pipeline.
  - Pre-release: what tool, which workload shape, what environment.
  - Production validation: what RUM signal would you instrument, and with what alert threshold.

  **Deliverable 4 — Failure-mode plan**

  Define:
  - What triggers a hard build block (vs. a warning).
  - How you distinguish a real regression from noise-floor variance. Name a specific threshold (e.g., >10% increase over 3-run median).
  - What escalation path a breached SLO follows if the regression is confirmed.

  **Deliverable 5 — Evidence**

  Run Lighthouse-CI on the target site once (locally or in CI) and report the actual numbers vs. your proposed budget. If the numbers breach your budget, name the specific Lighthouse diagnostic you would read first and what it would tell you.

  Rubric (revealed after submission):
  - Does each SLO have a percentile AND a workload shape AND a cache state? Missing any one is partial credit only.
  - Does the rationale cite an external anchor (CrUX, Google threshold, benchmark) rather than an internally-generated number? Unsupported numbers fail.
  - Does the test plan name a specific executor type for the backend load test (closed vs. open) and justify the choice?
  - Does the failure-mode plan define a noise-floor threshold (not just "we'll decide")? A threshold-free plan is a fail.
  - Does the evidence section report actual Lighthouse numbers, not "I would run Lighthouse"? Hand-waving fails.
  - Bonus: does the candidate name a failure mode the chosen workload shape will *not* catch, and name which alternative shape would catch it?
</PracticeTask>
