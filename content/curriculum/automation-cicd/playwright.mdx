## Core Idea

Playwright becomes reliable only when a tester learns it as a small set of **orthogonal primitives**, not as Selenium-with-different-syntax. The load-bearing primitives are: **Locator** (lazy, retryable DOM handle), **auto-wait** (five conditions checked before every action), **web-first assertion** (retries until match or timeout), **fixture** (composable scoped setup/teardown), **browser context** (isolated storage inside one process), and **trace** (time-travel artefact of an entire run).

The companion claim: flake is almost always a *primitive misuse*, not a Playwright bug. Race conditions, wrong assertion family, shared context state, missing auto-wait semantics вАФ each maps to a primitive the tester hadn't internalised. The trace converts "this flakes sometimes" into "here is the millisecond it broke."

> Flake is a primitive-misuse diagnostic, not an environmental excuse. The trace is the evidence that makes diagnosis possible.

## Worked Example

This site's `tests/e2e/concept-gate.spec.ts` uses several primitives at once вАФ it is the best anchor for understanding how they compose.

**What the spec does:** It tests the ConceptGate component on the `/projects/flaky-test-hunter` page вАФ a React island that hides project content until prerequisite cards are reviewed. It requires a mocked OAuth sign-in before navigating, uses `getByTestId` locators on dynamic content, and calls `@axe-core/playwright` for a11y verification in the same run.

**The sign-in helper вАФ `page.request` as a fixture-free API client:**

```ts
// tests/e2e/concept-gate.spec.ts (excerpt)
async function signInWithMockedGitHub(page: Page): Promise<void> {
  const startRes = await page.request.post('/api/auth/sign-in/social', {
    data: { provider: 'github', callbackURL: '/' },
    headers: { 'content-type': 'application/json' },
  });
  expect(startRes.ok(), `sign-in init failed: ${startRes.status()}`).toBeTruthy();
  const body = (await startRes.json()) as { url?: string };
  const state = new URL(body.url!).searchParams.get('state');

  const cbRes = await page.request.get(
    `/api/auth/callback/github?code=fake-e2e-code&state=${state}`,
    { maxRedirects: 0 },
  );
  expect(cbRes.status(), `callback status: ${cbRes.status()}`).toBeLessThan(400);
}
```

`page.request` shares the browser's cookie jar вАФ the resulting session cookie is visible to subsequent `page.goto()` calls without any `storageState` file. This is a test-scope auth pattern; for worker-scope auth, the equivalent is `globalSetup` writing a `storageState` file and each test loading it via `use: { storageState: 'auth.json' }`.

**The gate assertion вАФ web-first vs raw:**

```ts
// Web-first: polls until the element is visible (up to 10 s)
const gate = page.getByTestId('concept-gate');
await expect(gate).toBeVisible({ timeout: 10_000 });   // вЬУ retries

// Raw (DON'T use for async UI state):
const text = await gate.textContent();
expect(text).toBeTruthy();                              // вЬЧ reads once, no retry
```

The `concept-gate` is a React island that hydrates asynchronously. The raw form reads `textContent()` before hydration completes and produces a silent flake. The web-first form retries until the element is genuinely visible.

**The `addLocatorHandler` pattern вАФ dismissing overlays:**

```ts
// nav.spec.ts вАФ registered before every test
await page.addLocatorHandler(page.locator('vite-error-overlay'), async (el) => {
  await el.locator('button').first().click();
});
```

`addLocatorHandler` is a Playwright primitive that intercepts any matching element that blocks an action вАФ without it, a Vite error overlay that appears mid-test causes the next `click()` to time out on the "receives events" auto-wait condition (the overlay is covering the target).

**The `getByRole` vs `getByTestId` decision in this codebase:**

`nav.spec.ts` uses `page.getByRole('button', { name: /Open menu/i })` for the hamburger toggle вАФ a role-based locator that survives button-text refactors and also functions as an a11y assertion (if the button lacks an accessible name, the locator returns nothing and the test fails with a clear message). `concept-gate.spec.ts` uses `page.getByTestId('concept-gate')` for the React island root вАФ correct because the element has no meaningful semantic role; `data-testid` is the right fallback when role is genuinely absent.

**The `auth-quiz-streak.spec.ts` storageState trap:** That spec calls `page.evaluate()` to inject `sessionStorage` directly and then reloads вАФ bypassing the UI flow for quiz state. This is the correct use of storage manipulation: it short-circuits expensive UI steps while leaving the critical code path (the `POST /api/quiz/attempts` network call) fully exercised. Contrast this with `storageState` (cookies/localStorage written at globalSetup): mutating auth-related storage inside a test invalidates the captured state for every subsequent test in the worker.

## Common Pitfalls

- **Reaching for `getByTestId` or CSS before trying `getByRole`.** Role-based locators are more durable (survive text/class refactors) and double as an a11y assertion. Fix: follow the hierarchy вАФ `getByRole` first, then `getByLabel`, `getByPlaceholder`, `getByText`, finally `getByTestId`. CSS and XPath are last resorts. This happens because CSS and XPath are familiar from other toolchains and feel concrete.

- **Using `expect(await locator.textContent()).toBe('Foo')` on async UI.** This reads the DOM once вАФ if the value hasn't updated yet, it fails silently. The form `await expect(locator).toHaveText('Foo')` retries until match or timeout. These two look nearly identical and behave oppositely under any UI that updates after first paint. Fix: use the web-first assertion form for any state that can change after navigation or action.

- **`page.waitForTimeout(ms)` to stabilise a flaky step.** Sleep-based waits silently break when underlying timing changes вАФ they increase wall-clock without improving reliability. Fix: replace with the appropriate `expect(locator).toBeVisible()` or `expect(locator).toHaveAttribute(...)` assertion and let auto-wait do the work.

- **Reusing captured `storageState` across tests that mutate auth state.** `globalSetup` logs in once and writes cookies to a file; subsequent tests re-hydrate from that file. A test that logs out or deletes the account invalidates the file for every remaining test in the worker. Fix: use per-test isolated accounts for flows that mutate session state; use worker-scope fixtures only for immutable shared resources.

- **Registering `page.route()` or `page.waitForResponse()` after the action that triggers them.** Routes registered after a navigation don't intercept the request; `waitForResponse` after a click may miss a response that arrived first. Fix: wrap both in `Promise.all` so the listener is registered before the action: `const [resp] = await Promise.all([page.waitForResponse('**/api/save'), page.click(btn)])`.

- **Leaving `page.pause()` in committed tests.** A stray `pause()` blocks CI workers indefinitely; the symptom is "the suite is hanging" with no timeout error. Fix: add a lint rule or CI grep that fails on committed `pause()` calls.

- **Committing visual baselines from a non-Linux machine.** Font rendering, antialiasing, and GPU rasterisation differ across macOS/Linux/Windows even with Playwright's bundled Chromium. This repo bans non-Linux baselines explicitly; the regeneration workflow is `workflow_dispatch update_visual_baselines=true` on `ci.yml`. Fix: treat the pinned `@playwright/test` version as part of the baseline вАФ bumping the version requires regenerating baselines in the same PR.
