## Brief

Build a resilient, cross-browser Playwright suite against
[SauceDemo](https://www.saucedemo.com) — Sauce Labs' publicly hosted e-commerce
practice application. You'll cover the complete purchase journey across three
browser engines, layer in accessibility and visual regression tests, and document
the engineering decisions that make the suite maintainable rather than fragile.

This is a portfolio artifact. It should read like something a senior QA engineer
would ship on a real project — not a tutorial exercise.

> **Drift notice.** SauceDemo is a static frontend served by Sauce Labs. It
> has been stable at the same version (`swag-labs-v1`) for several years and
> is not expected to change. However, it is externally hosted; if the site is
> unavailable, use the
> [open-source fork](https://github.com/saucelabs/sample-app-web) which you can
> run locally with `npx serve dist`. Note in your README which URL you targeted
> and whether you ran locally or against the live site.

## Suggested approach

### Phase 1 — Single-browser foundation (Day 1)

1. Set up a clean Playwright project with TypeScript config.
2. Identify the two primary journeys: **happy-path purchase** (login → add to cart → checkout → order confirmation) and **failed login** (locked-out user, empty credentials).
3. Write page objects for `LoginPage`, `InventoryPage`, `CartPage`, and `CheckoutPage`.
4. Get everything green on Chromium before touching multi-browser.

### Phase 2 — Cross-browser (Day 2)

5. Add Firefox and WebKit to `playwright.config.ts`.
6. Run locally. Fix any engine-specific selector or timing issues. For each fix, add a comment explaining *why* the engine behaved differently — this becomes your postmortem material.
7. If a test is legitimately untestable on a given engine (e.g. a Chromium-only API), skip it with `test.skip('reason')` — not with `.only` or by removing it.

### Phase 3 — A11y + visual layers (Day 3)

8. Add `@axe-core/playwright`. Run `checkA11y` on the inventory page and the checkout confirmation. Capture any violations — you don't need to fix SauceDemo's a11y bugs, but document them.
9. Add two visual snapshots: the product inventory page (baseline) and the order confirmation page. Commit the baselines on Linux/CI if possible; document the OS you used.
10. Note: visual snapshots are font-sensitive across OS. If you can't run on Linux, commit the local baselines but add a note in the README and set `maxDiffPixelRatio: 0.05` to absorb rendering noise.

### Phase 4 — CI + resilience (Days 4–5)

11. Configure GitHub Actions to run all three browsers in parallel (matrix strategy or Playwright's built-in sharding).
12. Upload the HTML report as an artifact.
13. Induce one flake deliberately (e.g. remove a `waitFor` or force a race). Document the symptom and the fix in `docs/flake-postmortem.md` or the README. This is the most educational part of the project.
14. Set `retries: 1` in config — but only after you've fixed the real flake. Retry is a safety net, not a fix.

## What "done" looks like

A public GitHub repo with:
- Green CI badge (all three browsers, all passing)
- `pages/` directory with typed page objects
- At least one axe scan and two visual snapshots
- `docs/flake-postmortem.md` or a README section documenting: symptom, root cause, fix, and why `retries` is not the answer
- A README covering: how to run locally, how CI is structured, and one design decision you'd make differently on a second pass

## Why this matters

Cross-browser, a11y, and visual regression are the three layers that separate
a junior Playwright user from an engineer who ships suites that survive a
year of product changes. Doing all three on one project — and documenting a
real flake — is the evidence that you've moved past "tests that pass when I
run them."

SauceDemo is the right target because it is stable enough to be a reliable
baseline but complex enough (state management across pages, multiple user
types) to surface real browser-engine differences.
