## Core Idea

Observability is the property of a system that lets you ask any question about its current and past behaviour **without shipping new code**. It is enabled by three primary signals — *logs, metrics, traces* — and consumed at the moments your test suite cannot help you: after deployment, under real load, with real users.

For the tester, observability is the production-truth feedback loop the test suite cannot provide. A bug report that says "checkout is slow sometimes" is a wild goose chase without telemetry; paired with a Grafana panel it becomes "p99 latency spiked at 14:32, caused by a slow query introduced in the last deploy." The same investigation that took days now takes minutes.

The discipline pairs with testing: tests catch what you can pre-ship; observability reveals what tests cannot exercise — the tail of real traffic, real devices, real concurrency.

> Observability is not a replacement for testing. It is the production oracle that proves whether the thing you tested actually holds at scale.

## Worked Example

### Scenario: a user reports a slow checkout at 14:32 UTC

The tester's instinct without observability: reproduce the bug locally, run the test suite, shrug when it passes.

The tester's instinct with observability:

**Step 1 — pull the structured logs** for `user_id=12345`, timestamp window `[14:30, 14:35]`.

```
# Good: structured log — queryable in seconds
{"event":"checkout.failed","user_id":12345,"reason":"timeout","service":"payment","duration_ms":4821,"trace_id":"abc123"}

# Bad: unstructured log — grep archaeology
[ERROR] Checkout failed for user 12345 after 4821ms
```

With structured logs, the query `event="checkout.failed" AND user_id=12345` returns the failure in one command. Without structure, you grep through gigabytes and hope.

**Step 2 — open the trace** for `trace_id=abc123` in the APM tool (Jaeger, Datadog APM, Honeycomb).

```
Request: POST /api/checkout  total: 4821ms
├── auth-service         32ms   ✓
├── inventory-service   120ms   ✓
├── payment-service    4512ms   ← 93% of total time
│   ├── validate-card   18ms   ✓
│   └── charge-card    4490ms  ← retried 3 times (rate-limited by Stripe)
└── order-service        40ms  (never reached — parent timed out)
```

The trace reveals the exact failure in 30 seconds: the payment service retried three times against a rate-limited third-party API. No amount of local testing would surface this — Stripe's rate limit is a production-only condition.

**Step 3 — check the metric panel** in Grafana for `payment_service_duration_p99` in the 14:30–14:40 window.

```
14:00  p99: 320ms   ← baseline
14:30  p99: 4800ms  ← spike begins
14:35  p99: 4900ms
14:40  p99: 290ms   ← self-corrected (Stripe rate limit window expired)
```

The metric gives you the blast radius: how many users were affected and for how long. The alert that should have fired: a burn-rate alert on the `checkout_success_rate` SLO.

**What the tester produces:** a bug report with trace ID, log query, metric screenshot, and a proposed fix (add exponential backoff with jitter on Stripe retries). This is a forensically complete bug report. Compare to "checkout was slow at 2pm."

### The cardinality landmine (metric anti-pattern)

A team proposes: `http_request_duration_seconds{path="/api/checkout", user_id=12345, region="eu-west", status="500"}`.

The problem: `user_id` has millions of distinct values. One metric with four tags becomes `paths × users × regions × statuses` time-series in the metrics backend. At 1M users, 50 paths, 5 regions, 10 status codes that is 2.5 billion time series — enough to crash Prometheus.

The fix: template the path (`/api/:resource`), remove `user_id` entirely (move high-cardinality debugging to traces), keep `region` (5 values) and `status_class` (`2xx`/`4xx`/`5xx` — 3 values). Result: 50 × 3 × 5 = 750 time series. Same question-answering power; 3 million times fewer series.

**Tester's contribution:** ask "what is the cardinality of each label?" before a new metric ships. This is a test-design question, not a metrics question.

## Common Pitfalls

- **Unstructured logs.** Writing `log.error("Payment failed for user " + id)` instead of `log.error({event:"payment.failed", user_id: id, reason: err.code})` makes every log investigation an archaeology exercise. Fix: mandate JSON-structured logging with consistent field names (`event`, `user_id`, `trace_id`, `duration_ms`) as a definition-of-done criterion. This happens because structured logging requires a library change and teams defer it until the first painful debugging session.

- **Tagging metrics by user ID.** High-cardinality labels (user IDs, session IDs, email addresses) blow up the metrics backend — Prometheus scales by time-series count, not by sample count. Fix: use low-cardinality labels only; move high-cardinality debugging to traces or structured log queries. This happens because developers reach for the most specific identifier without understanding the cost model.

- **Alerting on internal thresholds, not user impact.** "CPU > 80%" fires constantly and is ignored; "SLO error budget exhausts in < 6 hours" fires rarely and is acted on. Fix: derive alerts from SLO burn rates, not from arbitrary resource thresholds. Page on symptoms users feel — errors, slow responses — not on internal state that may not correlate with user pain.

- **Head-based trace sampling discards the interesting traces.** 1% random sampling at 10k RPS keeps 100 traces/second — but the errored trace or the 5-second outlier has only a 1% chance of surviving. Fix: use tail-based sampling (keep the trace based on outcome — error, latency threshold — not at ingestion time). This happens because most systems default to head-based sampling for simplicity.

- **No correlation IDs across services.** A request that touches five services produces five log streams with no shared key, making cross-service debugging manual archaeology. Fix: propagate `trace_id` (or `request_id`) through every service call, in both logs and HTTP headers, via OpenTelemetry context propagation. Without it, connecting a frontend error to the backend failure that caused it is guesswork.

- **Forgetting source maps on deploy.** A minified production JS error (`at t.e (main.abc123.js:1:4820)`) is unreadable without the original source. Fix: upload source maps to your error tracker (Sentry, Datadog RUM) as part of every deploy step. This happens because source maps are a build concern and teams don't wire them into the deploy pipeline until the first unreadable production stack trace.

- **PII and secrets in logs.** Full request bodies, auth tokens, or email addresses in log lines are an audit finding and a breach risk. Fix: use field-level redaction at the logging library level (pino's `redact` option; structured loggers all support it). This happens because developers log "the whole object" for debugging convenience and never remove it.
